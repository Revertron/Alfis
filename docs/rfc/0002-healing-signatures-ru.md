# RFC-0002: Самовосстановление цепочки через резервный пул подписей

| Поле       | Значение                                              |
|------------|-------------------------------------------------------|
| Номер RFC  | 0002                                                  |
| Название   | Самовосстановление цепочки через резервный пул подписей |
| Статус     | Черновик                                              |
| Тип        | Standards Track (изменение консенсуса)                |
| Создан     | 2026-07-11                                            |
| Авторы     | Проект ALFIS                                          |
| Связан с   | RFC-0001 (Аварийное восстановление подписантов)       |

## Аннотация

Каждый полный блок ALFIS должен быть зафиксирован как минимум
`BLOCK_SIGNERS_MIN = 4` подписывающими блоками из набора в 7 ключей,
выбираемых псевдослучайно и навсегда фиксируемых собственной подписью
полного блока. Если менее четырёх из выбранных ключей работоспособны,
цепочка останавливается навсегда: нет ни таймаута, ни повторного
выбора, ни альтернативного пути подписания.

Этот RFC вводит **лечащие подписи** (healing signatures): после того
как полный блок остаётся незафиксированным в течение `HEALING_TIMEOUT`
(пять дней), недостающие подписи — до тех же четырёх в сумме — могут
быть добавлены детерминированным **резервным пулом** из не более чем
двенадцати ключей, выводимых из истории цепочки: семь самых
продуктивных ключей за всё время (*якорный пул*) плюс пять самых
активных ключей недавнего окна, не входящих в якорный пул (*недавний
пул*). Если в фиксации участвуют лечащие подписи, как минимум одна из
учитываемых подписей должна принадлежать якорному ключу.

Лечащие подписи — это обычные пустые подписывающие блоки. Механизм не
добавляет ни новых типов сообщений, ни новых классов транзакций, ни
конфигурации, ни криптографии; изменение консенсуса ограничено
правилами валидности подписи и полноты фиксации. В отличие от
RFC-0001, он не требует церемонии ключей и активируется автоматически,
оставаясь намеренно консервативным: он *дополняет* существующую
фиксацию четырьмя подписями, а не заменяет её другим кворумом.

Дополнительно этот RFC делает нормативными три исправления
существующего механизма фиксации, от которых зависит лечащее правило
(раздел 3.7).

## 1. Мотивация

### 1.1 Сценарий остановки

Семь допустимых подписантов полного блока `F` выбираются
псевдослучайным обходом с зерном из хвоста `F.signature` по множеству
исторических авторов блоков, проходящих фильтр
`min_block_count = height / 100` (см. `get_block_signers` в
`src/blockchain/chain.rs`). Два свойства в сумме создают риск
постоянной остановки:

1. Набор подписантов для `F` фиксируется в момент добычи `F`. Если
   менее четырёх из выбранных ключей когда-либо произведут
   подписывающий блок, `F` навсегда останется незафиксированным, и
   следующий полный блок будет запрещён.
2. Фильтр допуска — храповик: на текущей высоте ключ должен иметь
   ~225 блоков, что на практике исключает новичков и замораживает пул
   на небольшом наборе ветеранов.

Измерено на снимке цепочки на высоте 22 524 (2026-07-05):

* 715 различных ключей когда-либо создавали блоки; лишь **16**
  проходят порог допуска.
* Топ-7 ключей по общему числу блоков имеют от 1 411 до 2 329 блоков
  каждый, подавляюще — подписывающих; все 16 допущенных ключей активны
  в последних 5 000 блоках, но примерно **половина самых продуктивных
  ключей контролируется одним оператором** (мейнтейнером проекта,
  владеющим распределённым парком узлов; остальные держатели — как
  правило, одиночные домашние машины).
* В том же снимке за последним полным блоком (индекс 22 521,
  2026-07-05 03:58:50 UTC) следуют ровно **три** подписывающих блока —
  на один меньше, чем нужно для фиксации, — и больше ни одного блока.
  Иллюстрация розыгрыша, четвёртая подпись которого может просто
  никогда не прийти.

Последствия «фактора автобуса» подробно разобраны в RFC-0001, раздел
1.2, и здесь не повторяются.

### 1.2 Почему не механизм пороговой подписи из RFC-0001

RFC-0001 решает катастрофический вариант проблемы (потеряна вся
популяция подписантов) с помощью аварийного ключа FROST 5-из-9,
хранимого настроенными держателями долей. Его стоимость значительна:
церемония распределённой генерации ключа, инструментарий и UI для
держателей, три новых типа gossip-сообщений с сессионным состоянием,
дисциплина периодического перераспределения долей и внешне
поддерживаемый список людей.

Для *обычного* варианта — часть выбранных подписантов утрачена,
большинство продуктивных ключей живо — достаточно куда меньшего
механизма, полностью переиспользующего существующую машинерию
подписывающих блоков. RFC-0001 остаётся страховкой для случая, который
этот RFC покрыть не может (одновременная потеря практически всего
резервного пула, см. раздел 7).

### 1.3 Ограничения дизайна, принятые по итогам предыдущего анализа

Пространство решений было сужено следующими осознанными решениями:

1. **Никакого ослабления кворума по времени** (например, расширения
   выбранной семёрки или снижения порога со временем). Такой распад
   активируется симметрично по обе стороны сетевого разделения и
   позволяет продвигаться обеим сторонам, порождая длинные
   несовместимые форки. Протокол должен по-прежнему предотвращать
   само *образование* форков.
2. **Никакого скользящего окна допуска для выбираемых подписантов.**
   В пул на основе недавней активности может войти атакующий, какое-то
   время намайнивший блоков и затем исчезнувший или ставший
   враждебным (усугубляется тем, что *продления* доменов освобождены
   от 24-часового кулдауна и получают скидку к сложности, т.е.
   активность подделывается дешевле, чем кажется).
3. **Лечащий набор должен выводиться из данных цепочки, а не из
   конфигурации.** Настроенный список гарантирует «один ключ — один
   человек», но требует церемонии и сопровождения как в RFC-0001;
   набор из цепочки не видит людей, зато самоподдерживается и
   бесплатен. Этот RFC принимает «слепоту к людям», смягчает её риски
   якорным правилом (раздел 3.5) и таймаутом и честно документирует
   остаточную концентрацию (раздел 7).
4. **Лечение дополняет существующую фиксацию четырьмя подписями**, а
   не образует отдельный мажоритарный кворум. Резервный пул — это
   запас прочности, а не параллельное правительство.

## 2. Терминология

| Термин | Определение |
|--------|-------------|
| Полный блок | Блок с непустой транзакцией (регистрация или продление домена). |
| Окно фиксации | Последовательность пустых блоков между полным блоком `F` и следующим полным блоком. |
| Выбранные подписанты | 7 ключей, выбранных для `F` функцией `get_block_signers(F)`. |
| Якорный пул `A(F)` | 7 ключей с наибольшим числом блоков в диапазоне `(1, F.index]`. |
| Недавний пул `R(F)` | 5 самых активных ключей последних `HEALING_WINDOW` блоков перед `F`, исключая членов `A(F)`. |
| Резервный пул `P(F)` | `A(F) ∪ R(F)`, не более 12 ключей. |
| Лечащая подпись | Пустой блок в окне фиксации `F`, созданный членом `P(F)` при условии таймаута из раздела 3.4. |
| Якорная подпись | Любой учитываемый подписывающий блок (выбранный или лечащий), автор которого входит в `A(F)`. |
| Вылеченное окно | Окно фиксации, содержащее хотя бы одну лечащую подпись. Распознаётся структурно: пустой блок, автор которого не входит в выбранных подписантов полного блока своего окна. |

## 3. Спецификация

### 3.1 Константы

```
HEALING_ACTIVATION_HEIGHT  = <задаётся при релизе; для остановленной цепочки: сам застрявший полный блок, см. 6.1>
HEALING_TIMEOUT            = 5 * 86400 секунд   (432 000)
HEALING_TIMEOUT_FAST       = 3 * 86400 секунд   (259 200)
HEALING_LOOKBACK           = 10 блоков
HEALING_POOL_ANCHORS       = 7
HEALING_POOL_RECENT        = 5
HEALING_WINDOW             = 5 000 блоков
HEALING_ANCHORS_MIN        = 1
```

Все правила этого RFC применяются только к блокам с
`index > HEALING_ACTIVATION_HEIGHT`. Историческая валидация не
меняется.

### 3.2 Вывод резервного пула

Для полного блока `F` каждый узел детерминированно вычисляет:

* **`A(F)`** — ранжировать все ключи по числу блоков, созданных ими в
  диапазоне `1 < index ≤ F.index`; взять первые
  `HEALING_POOL_ANCHORS`. Ничьи разрешаются по возрастанию
  лексикографического порядка байтов публичного ключа.
* **`R(F)`** — ранжировать все ключи, **не входящие в `A(F)`**, по
  числу блоков в диапазоне
  `max(1, F.index − HEALING_WINDOW) < index ≤ F.index`; взять первые
  `HEALING_POOL_RECENT`, ничьи — так же. Если различных неякорных
  ключей в окне меньше, `R(F)` просто меньше.
* **`P(F) = A(F) ∪ R(F)`**.

Ранжирование учитывает блоки **всех** классов (полные и подписывающие).
Оба ранжирования — функции цепочки до `F.index` включительно, поэтому
каждый узел вычисляет один и тот же пул независимо от собственной
вершины, а пул конкретного окна фиксации неизменен с момента появления
`F`. Результат СЛЕДУЕТ кэшировать по `F.index` по аналогии с
существующим `SignersCache`.

### 3.3 Валидность лечащей подписи

Пустой блок `E` в окне фиксации полного блока `F` является валидным
подписывающим блоком, если **либо**:

* `E.pub_key` входит в выбранных подписантов `F` (существующее
  правило, без изменений), **либо**
* `E.pub_key ∈ P(F)` **и** для `E` выполнено условие таймаута из
  раздела 3.4.

Все прочие правила пустых блоков действуют без изменений в обоих
случаях: proof-of-work не ниже `SIGNER_DIFFICULTY`; корректный хэш и
подпись Ed25519; таймштамп не раньше предшествующего блока и не более
чем на 60 секунд в будущем на момент получения; **один блок на ключ на
окно фиксации** (существующий обход в `is_good_signer_for_block`);
автор самого `F` подписывать не может. Дополнительно, начиная с
`HEALING_ACTIVATION_HEIGHT`, ключ автора пустого блока должен
удовлетворять `KEYSTORE_DIFFICULTY` — закрывается пробел, при котором
стойкость ключа проверялась только у полных блоков.

Ключ, являющийся одновременно выбранным подписантом и членом пула, —
просто выбранный подписант; лечащий путь для него ничего не меняет
(одна подпись в любом случае).

### 3.4 Условие таймаута

Таймаут, применимый к полному блоку `F`, адаптивен:

```
T(F) = HEALING_TIMEOUT_FAST   если любой из HEALING_LOOKBACK блоков,
                              предшествующих F (индексы F.index-10 ..
                              F.index-1), является лечащей подписью
       HEALING_TIMEOUT        иначе
```

Является ли прошлый блок лечащей подписью, решается чисто структурным
тестом — пустой блок, автор которого не входит в выбранных
подписантов полного блока своего окна (никакой другой блок не может
валидно иметь такую форму, так что поле-маркер не нужно). Результат по
окну СЛЕДУЕТ кэшировать.

Лечащая подпись `E` для полного блока `F` тогда валидна, только если:

```
E.timestamp - F.timestamp >= T(F)
```

Точка отсчёта — **таймштамп самого полного блока**, а не вершины,
чтобы (а) условие было чистой функцией данных блоков и могло быть
перепроверено оффлайн байт-в-байт и (б) каждый принятый лечащий блок
не перезапускал часы для следующего: как только окно «протухло» на
`T(F)`, все члены пула квалифицируются одновременно.

Условие нельзя выполнить досрочно подделкой: таймштампы блоков
отклоняются, если опережают часы принимающего узла более чем на 60
секунд, и обязаны быть монотонно неубывающими вдоль цепочки.
Единственный способ получить валидный лечащий таймштамп — чтобы пять
дней действительно прошли на публично наблюдаемой остановленной
цепочке.

### 3.5 Полнота фиксации (якорное правило)

Пусть `S` — множество валидных подписывающих блоков в окне фиксации
`F` (каждый блок валиден по разделу 3.3, все авторы различны). Блок
`F` **зафиксирован** тогда и только тогда, когда:

```
|S| >= BLOCK_SIGNERS_MIN (4)
И ( S содержит не менее BLOCK_SIGNERS_MIN выбранных подписей
    ИЛИ как минимум HEALING_ANCHORS_MIN блок(ов) из S создан(ы)
        членом A(F) )
```

Словами: четыре выбранные подписи фиксируют всегда, ровно как сегодня
— случайная лечащая подпись в окне не может поднять планку для
жеребьёвочной фиксации (иначе один резервный ключ мог бы *отравить*
медленное, но восстановимое окно, подбросив в него одну лечащую
подпись и потребовав якорь, который жеребьёвочной фиксации никогда не
нужен). Фиксация же, которой лечащие подписи нужны, чтобы добрать до
четырёх, должна включать хотя бы одну подпись — выбранную *или*
лечащую — от ключа из топ-7 за всё время. Заметим, что в редких
комбинациях это может потребовать пятого подписывающего блока
(например, три выбранные подписи от неякорных ключей плюс одна
лечащая из недавнего пула фиксацию не дают; окно остаётся открытым,
пока не подпишет якорь или не придёт четвёртая выбранная подпись).

Предикат фиксации заменяет текущую арифметику разности высот
(`sign_count = height - full.index`) в `is_waiting_signers`,
`next_allowed_full_block`, `get_sign_block` и в валидации полных
блоков. Обход покрывает не более `BLOCK_SIGNERS_ALL` блоков и дёшев.

### 3.6 Поведение майнера

Узел, владеющий ключом из `P(F)`, обрабатывает возможность лечения
ровно как возможность подписания: `get_sign_block` дополнительно
предлагает лечащую задачу, когда локальные часы показывают, что
таймаут истёк, цепочка не синхронизируется, фиксация не завершена и
ключ ещё не подписывал это окно. Действует существующая случайная
задержка старта (`BLOCK_SIGNERS_START_RANDOM`, до 90 с); для лечащих
задач её СЛЕДУЕТ увеличить (предложение: до 15 минут), поскольку
лечение никогда не критично по задержке, а больший разброс сокращает
бесполезный параллельный майнинг среди до двенадцати участников.

Узел предлагает только те подписи, которые способны продвинуть
фиксацию: когда четыре подписи уже есть, но окно всё ещё не
зафиксировано (раздел 3.5), лечащие задачи берут только якорные ключи
— ещё одна не-якорная лечащая подпись ничего не продвинет, тогда как
выбранные подписи продвигают всегда (они идут в счёт чисто
жеребьёвочной фиксации).

Участие автоматическое. Никаких действий оператора, UI или
конфигурации не требуется. Узлы без ключей пула ведут себя ровно как
сегодня.

### 3.7 Исправления-ужесточения (нормативные)

Лечащее правило имеет смысл, только если фиксация действительно
проверяется. Три дефекта текущей реализации исправляются начиная с
`HEALING_ACTIVATION_HEIGHT`:

1. **Проверять фиксацию у принимаемых полных блоков.**
   `is_good_sign_block` сейчас возвращает `true` для любого полного
   блока до достижения проверки, отклоняющей полный блок поверх
   незафиксированного предшественника (мёртвый код с 2021 года, коммит
   `f8d47df`). С момента активации полный блок, чей предшествующий
   полный блок не зафиксирован (по разделу 3.5), невалиден.
2. **Закрыть пробел «после седьмой подписи».** Пустой блок поверх
   полного блока, который уже **зафиксирован** и имеет
   `BLOCK_SIGNERS_ALL` (7) подписывающих блоков, невалиден. (Сейчас
   после появления семи подписывающих блоков принимается сколько
   угодно дальнейших пустых блоков от любого ключа.) Пустые блоки с
   пятого по седьмой над зафиксированным окном остаются валидными, как
   и сегодня, — для устойчивости к гонкам подписания. *Незафиксированное*
   же окно принимает любой валидный подписывающий блок независимо от их
   числа. Эта оговорка принципиальна: безусловный потолок в семь блоков
   мог бы навсегда заблокировать цепочку — три жеребьёвочные подписи
   не-якорей плюс четыре лечащие подписи недавнего пула заполняют семь
   мест, не удовлетворив якорному правилу, и якорная подпись, которая
   завершила бы фиксацию, должна оставаться допустимой. Окно при этом
   всё равно ограничено: один блок на каждый жеребьёвочный или
   резервный ключ.
3. **Стойкость ключа для пустых блоков**, как указано в разделе 3.3.

Дополнительно (вне консенсуса, но обязательное поведение): узел,
обнаруживший пира с тем же origin, отдающего цепочку, несовместимую с
его собственной (форк глубже `LIMITED_CONFIDENCE_DEPTH`), ОБЯЗАН
сообщить об этом громко — отдельной строкой лога уровня `error` и
событием для UI — вместо нынешнего молчаливого бана. Сбои
воссоединения после разделения сети должны быть видимы людям.

## 4. Обоснование

### 4.1 Почему дополнение до четырёх, а не большинство пула

Мажоритарный порог по резервному пулу (например, 7-из-12) был бы
эксклюзивен при разделении сети по принципу Дирихле — большинство
могла бы собрать не более чем одна сторона, — но лишь при условии, что
ключи пула соответствуют разным людям. Это не так: цепочка видит
ключи, а не людей, и сегодня примерно половина якорного пула — один
оператор. Правило большинства ключей купило бы *видимость*
децентрализованной эксклюзивности без её сущности, усложнив при этом
восстановление в каждом честном сценарии. Вместо этого RFC сохраняет
привычную фиксацию четырьмя подписями, а историю про разделение сети
выводит из топологии и якорного правила (раздел 4.4).

### 4.2 Почему якоря ранжируются за всё время

В ранжирование за всё время нельзя войти быстро: чтобы обогнать ключи
с 1 400–2 300 блоками, нужно перепроизвести пять лет непрерывной
работы. Это та устойчивость к «накрутке», которой лишён недавний пул.
Якорный набор дрейфует — медленно, годами — по мере того как живые
ключи накапливают блоки, так что он не заморожен навсегда, как
храповик допуска `height/100`, но захватить его в пределах одного окна
атакующих усилий невозможно.

### 4.3 Зачем вообще нужен недавний пул

Одни лишь якоря воспроизводят проблему смертности уровнем выше: в
день, когда держатели якорей уйдут, лечение умрёт вместе с ними.
Недавние слоты — это путь входа: любой ключ, ставший по-настоящему
активным — включая чисто доменные ключи, которым храповик допуска
никогда не даст стать подписантами, — попадает в `R(F)` в пределах
одного окна и добавляет лечащую ёмкость. По замерам на текущей
вершине, `R(F)` состоял бы из двух ветеранских подписантских ключей
сразу за якорной чертой (1 326 и 1 213 блоков всего) и трёх умеренно
активных ключей (229–436 блоков всего), один из которых — почти
исключительно доменный. Это свидетельство того, что путь входа
принимает участников, структурно недоступных системе выбранных
подписантов.

### 4.4 Зачем якорное правило

Недавний пул поддаётся накрутке (раздел 1.3, пункт 2), а ключ — в
отличие от честного человека — может подписывать по *обе* стороны
разделения, если его владелец намеренно сидит на обеих. Без якорного
правила атакующий, купивший четыре недавних слота во время настоящей
остановки, мог бы вылечить обе стороны разделения, устроенного им же
на сетевом уровне, — изготовив ровно тот долговременный форк, ради
предотвращения которого всё и затевалось. С `HEALING_ANCHORS_MIN = 1`
никакая комбинация подписей недавнего пула нигде не зафиксирует блок
без соподписи хотя бы одного из семи медленно меняющихся, не
поддающихся накрутке ключей. Лечение по обе стороны разделения
требует, следовательно, участия якорей на обеих сторонах — либо двух
честных держателей якорей, действительно разделённых сетью, либо
скомпрометированного якорного ключа.

Повышение параметра до 2 усилило бы защиту ценой хрупкости
восстановления: при сегодняшней концентрации требование двух якорей
после ухода мажоритарного оператора могло бы означать требование
*обоих* оставшихся независимых якорных ключей. Один якорь — осознанный
баланс; параметр существует, чтобы будущий релиз мог скорректировать
его по мере децентрализации якорного набора.

### 4.5 Почему пять дней — и три в деградированном режиме

Базовый таймаут служит трём целям: он держит лечащий путь строго
подчинённым нормальному подписанию (здоровый кворум фиксирует за
минуты); даёт людям время заметить остановку по обычным каналам до
того, как сработает протокол; и является главной защитой от остаточной
геометрии двойного лечения (раздел 5.2) — разделение должно
продержаться дольше `HEALING_TIMEOUT` *и* иметь лечащую ёмкость по обе
стороны, прежде чем расхождение станет возможным. Пять дней — сильно
больше любого правдоподобного безобидного сбоя, но достаточно мало,
чтобы владельцы доменов в 30-дневном льготном окне продления потеряли
немного.

Однако если потеря подписантов реальна и постоянна, лечится *каждое*
окно, и фиксированное пятидневное ожидание на окно душит цепочку до
~6 доменов в месяц, ничего не покупая: работа по безопасности, которую
выполняет длинный таймаут — доказательство подлинности остановки людям
и протоколу, — уже оплачена первым лечением. Отсюда адаптивное правило
раздела 3.4: как только цепочка доказуемо работает в вылеченном режиме
(лечащая подпись в последних `HEALING_LOOKBACK` блоках, т.е. примерно
в последних двух окнах фиксации), последующие окна ждут лишь три дня.

В быстрый путь нельзя войти синтетически: чтобы «лечение попало в
окно просмотра», требуется честно вылеченное окно, а оно требует
настоящей пятидневной остановки плюс якорной подписи. На любой ветви,
которая была здорова в момент разделения, первое лечение поэтому
всегда платит полные пять дней, так что шлюз двойного лечения из
раздела 5.2 не меняется для разделений здоровых цепочек; лишь цепочка,
*уже* находившаяся в деградированном режиме в момент разделения,
стартует с трёх дней на обеих сторонах. Возврат автоматичен: два
подряд нормально зафиксированных окна выталкивают все лечащие подписи
из окна просмотра, и таймаут возвращается к пяти дням. Эскалирующая
шкала (дальнейшее сокращение после многих подряд вылеченных окон)
рассматривалась и отброшена ради простоты.

### 4.6 Анализ разделений сети

Рассмотрим реалистичные геометрии разделения при нынешней топологии
(один оператор с распределённым мультисайтовым парком держит примерно
половину резервного пула; остальные держатели — одиночные домашние
машины):

* **Парк гаснет целиком.** Это сбой, а не разделение: погасшие узлы
  ничего не строят. Выжившая сторона лечится через пять дней силами
  независимых якорей и недавних ключей; вернувшийся парк просто
  отстал и чисто досинхронизируется. Это основной сценарий «фактора
  автобуса», и механизм обрабатывает его без всякой координации.
* **Разделение изолирует парк от всех остальных.** Сторона парка
  лечится единолично. Другая сторона лечится, только если содержит
  якорный ключ *и* четыре подписи пула в сумме; иначе она
  останавливается, её расходящийся хвост остаётся в пределах
  `LIMITED_CONFIDENCE_DEPTH` (незафиксированный полный блок плюс не
  более трёх подписей — честные узлы никогда не строят поверх
  незафиксированного блока), и существующая машинерия Fork/lagged
  автоматически поглощает его при воссоединении.
* **Обе стороны держат якоря и ёмкость пула, а разделение переживает
  таймаут.** Обе стороны лечатся, образуется долговременный форк. Это
  остаточный риск, ограниченный якорным правилом, таймаутом и
  маловероятностью чистого долгоживущего разделения, отделяющего
  намеренно мультихоумный парк от четырёх кооперирующихся независимых
  держателей. Он делается *видимым* обязательной тревогой о форке из
  раздела 3.7.

### 4.7 Чего лечение намеренно не делает

Лечение добавляет пустые блоки. Оно не может зарегистрировать,
передать или отнять домен (все правила идентичности не тронуты); не
может переписать историю (правило `LIMITED_CONFIDENCE_DEPTH` не
тронуто); не может добывать полные блоки от резервных ключей с
какими-либо особыми правами; и не меняет выбор подписантов для
последующих блоков. Вся его власть — завершить фиксацию, которую
выбранные подписанты завершить не смогли, с опозданием в пять дней.

## 5. Соображения безопасности

### 5.1 Накрутка в недавний пул

Вход в `R(F)` требует перепроизвести текущие активные неякорные ключи
в окне из 5 000 блоков. Поскольку продления доменов освобождены от
24-часового кулдауна и получают скидку к сложности, производство
блоков ради активности дешевле, чем предполагает кулдаун (этот пробел
существует независимо от данного RFC и заслуживает отдельного
исправления). Якорное правило ограничивает выгоду: накрученные ключи
могут вносить лечащие подписи, но никогда не завершат фиксацию без
якоря, так что атака сама по себе не способна ничего разморозить — и
тем более ничего форкнуть. Остаточный эффект накрутки — вытеснение:
выталкивание честных ключей из `R(F)` сокращает честную лечащую
ёмкость максимум на пять подписей. Якорей плюс одного честного
недавнего ключа по-прежнему достаточно.

### 5.2 Двойное лечение при разделении сети

Разобрано в разделе 4.6. Требования для долговременного форка:
разделение дольше `HEALING_TIMEOUT` (или `HEALING_TIMEOUT_FAST`, если
цепочка уже была в вылеченном режиме на момент разделения — см. раздел
4.5), четыре подписи пула на каждой
стороне, якорь на каждой стороне и настоящая остановка (таймаут нельзя
спровоцировать; нужен фактический отказ выбранных подписантов). Каждый
фактор по отдельности правдоподобен; их совокупность — принятый
остаточный риск, и он деградирует изящно: тревога о форке превращает
его в громкий операционный инцидент, а не в молчаливый постоянный
раскол.

### 5.3 Компрометация ключей пула

Один скомпрометированный ключ пула (якорный или недавний) даёт не
более одной подписи на окно фиксации и лишь после пятидневной
публичной остановки. Четыре скомпрометированных ключа пула, включая
якорь, могут лечить единолично — та же способность, что у честного
пула, — но не получают ни одной из возможностей, перечисленных в
разделе 4.7. Ценность кражи резервных ключей поэтому низка; главная
защита в том, что якоря по построению — ключи операторов, годами
державших инфраструктуру.

### 5.4 Спам и DoS

Лечащие блоки стоят proof-of-work `SIGNER_DIFFICULTY`, ограничены
одним на ключ на окно фиксации, валидны лишь после пяти дней
остановки, а окно допускает не более одного блока на каждый
жеребьёвочный или резервный ключ — и не более семи в сумме после
фиксации (раздел 3.7, пункт 2). Никакого сессионного состояния на
уровне gossip для флуда нет; сообщения — обычные блоки.

### 5.5 Взаимодействие с RFC-0001

Если в итоге будут развёрнуты оба RFC, лечение — первая линия
(автоматическая, покрывает частичную потерю подписантов), а аварийный
блок RFC-0001 — вторая (управляемая людьми, покрывает потерю
практически всего пула). Их предусловия естественно упорядочены:
лечение на пятый день, аварийный механизм на седьмой. Цепочка,
вылеченная этим RFC, никогда не доживает до предусловия RFC-0001.

## 6. Обратная совместимость

Узлы, не реализующие этот RFC, будут отклонять лечащие блоки как
имеющие неверного подписанта, а также каждый последующий блок,
построенный на них. Все операторы узлов ОБЯЗАНЫ обновиться, чтобы
продолжить следовать за цепочкой, когда начнётся лечение. Релизу,
поставляющему этот RFC, СЛЕДУЕТ
поднять минимальную версию пиров (`version_compatible` в
`src/p2p/network.rs`), чтобы устаревшие узлы исключались из роя, а не
молча ответвлялись.

Ни один исторический блок не пересуживается: все правила, включая
исправления раздела 3.7, применяются только выше высоты активации.
`CHAIN_VERSION` не меняется; единственный переключатель — высота
активации.

### 6.1 Активация на уже остановленной цепочке

Обычно высота активации выбирается выше текущей вершины с щедрым
запасом, чтобы операторы успели обновиться до того, как правила смогут
сработать. Остановленная цепочка ломает эту схему: высота больше не
растёт, так что высота выше вершины никогда не будет достигнута и не
даёт никакого запаса.

Решение — установить `HEALING_ACTIVATION_HEIGHT` равной индексу самого
незафиксированного полного блока (22 521 для текущей остановки), чтобы
новые правила управляли ровно тем окном фиксации, которое застряло.
Запас не нужен, потому что остановка — тот единственный момент, когда
изменение консенсуса безопасно по построению:

* По до-RFC правилам цепочку продлить нельзя — это и есть остановка.
  Необновлённые узлы, следовательно, не могут построить конкурирующую
  ветвь. Они отклоняют лечащие блоки, замирают на вершине остановки и
  продолжают обслуживать своё замороженное состояние, пока оператор
  не обновится — после чего просто перематываются вперёд на
  вылеченную цепочку, строгим префиксом которой является их
  собственная. Форкать нечего.
* Если недостающая выбранная подпись неожиданно придёт *до* лечения,
  цепочка расклинится сама, и новые правила останутся спящими. Если
  *после* — она будет конкурировать с лечащим блоком на одном индексе,
  и существующее правило форков `is_better_than` разрешит спор.

Окно обновления, таким образом, — вопрос операционный, а не вопрос
безопасности консенсуса. Но коротким оно быть всё же должно, потому
что остановка активно уничтожает ценность: домены, чей 30-дневный
льготный срок продления истекает во время остановки, безвозвратно
уходят из рук владельцев. По замерам текущей цепочки (1 391 домен
зарегистрирован за всю историю, 594 всё ещё резолвятся на дату
остановки): ~22 домена выпадают из льготного окна к 2026-09-01, ~57 к
2026-10-01, ~213 к 2027-01-01. Повышение минимальной версии пиров
(выше) служит заодно и будильником: замороженные узлы активно
отключаются от роя, вместо того чтобы вечно простаивать на цепочке,
которая больше никогда не сдвинется.

## 7. Ограничения: запас хода, а не бессмертие

Резервный пул берётся из той же смертной популяции ключей, что и
выбранные подписанты. Этот RFC существенно расширяет пережимаемый
цепочкой бюджет потери ключей — любые четыре ключа пула с якорем среди
них сохраняют цепочку живой, — но не делает её бессмертной:

* Если популяция выбранных подписантов вымрет массово, а пул
  сохранится, каждое окно фиксации будет выжидать таймаут, и
  пропускная способность цепочки деградирует примерно до одного
  полного блока на `HEALING_TIMEOUT_FAST` (три дня — после
  установления деградированного режима, раздел 3.4). Это выживание,
  а не здоровье.
  (Каждое вылеченное окно добавляет блоки в счёт лечивших, медленно
  смещая `A(F)` к живым ключам, но храповик допуска выбранных
  подписантов вне рамок этого RFC и продолжит выбирать из того же
  исторического пула.)
* Если распадётся сам якорный пул — до точки, где не работает ни один
  якорь, — лечение перестаёт работать полностью, и цепочка
  возвращается к до-RFC сценарию отказа. При текущей концентрации
  уход мажоритарного оператора оставляет два независимых якорных
  ключа; потеря и их тоже исчерпывает механизм. Это точка, где
  остаётся путь RFC-0001 (или последующий хардфорк).

Эти пределы приняты осознанно: альтернативные дизайны, снимающие их
(скользящее окно допуска подписантов, настроенные списки людей,
пороговая криптография), были отвергнуты каждый по причинам из раздела
1.3 и по профилю стоимости самого RFC-0001.

## 8. Референсная реализация

Ожидаемая форма (всё в существующих файлах, без новых модулей):

1. `src/commons/constants.rs`: восемь констант раздела 3.1.
2. `src/blockchain/chain.rs`:
   * `get_healing_pool(&self, full_block: &Block) -> HealingPool`
     (якорные и недавние ключи плюс таймаут окна) с кэшем по аналогии
     с `SignersCache`; два SQL-ранжирования с лексикографическим
     разрешением ничьих.
   * `is_locked(&self, full_block: &Block) -> bool`, реализующий
     предикат раздела 3.5; используется в `is_waiting_signers`,
     `next_allowed_full_block` и в валидации полных блоков
     (исправление 3.7.1).
   * `is_good_signer_for_block`: принимать членов пула при условии
     таймаута (разделы 3.3/3.4); проверять стойкость ключа у пустых
     блоков; отклонять лишние пустые блоки над зафиксированным окном
     (исправление 3.7.2).
   * Распознавание вылеченных окон для адаптивного таймаута (пустой
     блок с автором вне набора выбранных подписантов), с кэшем по
     окну.
   * `get_sign_block`: предлагать лечащие задачи (раздел 3.6).
3. `src/p2p/network.rs` / событие UI: тревога о форке из раздела 3.7.
4. Тесты: детерминизм вывода пула (включая ничьи), граница таймаута,
   комбинации якорного правила (3 выбранные + 1 недавняя НЕ фиксируют;
   3 выбранные + 1 якорная фиксируют; 4 недавние НЕ фиксируют;
   3 недавние + 1 якорная фиксируют), устойчивость к гонкам для блоков
   5–7, отсечка по высоте активации на исторической БД
   `tests/blockchain.db`.

## 9. Открытые вопросы

1. Параметры таймаута: базовый `HEALING_TIMEOUT` (предложено пять
   дней; защитимы 3–7 — семь совпало бы с предусловием RFC-0001),
   `HEALING_TIMEOUT_FAST` деградированного режима (предложено три
   дня) и `HEALING_LOOKBACK` (10 блоков ≈ последние два окна
   фиксации; большее значение держит быстрый путь открытым при
   редких нормальных фиксациях в преимущественно деградированной
   цепочке).
2. `HEALING_ANCHORS_MIN`: 1 (предложено) или 2 (сильнее против
   накрутки и «сидения на двух стульях», хрупче для преемственности).
3. Должен ли `R(F)` ранжироваться только по подписывающим блокам. Это
   сделало бы недавний пул недоступным для накрутки (подписывать можно
   только будучи выбранным, а выбор требует допуска), но закрыло бы
   путь входа для доменных ключей — вернув ту самую закрытость, ради
   разрушения которой недавний пул существует. Предложено: ранжировать
   по всем блокам, полагаться на якорное правило.
4. Исправлять ли пробел кулдауна продлений (продления освобождены от
   `NEW_DOMAINS_INTERVAL`) в том же релизе. Рекомендуется, поскольку
   это удорожает накрутку в целом, но это независимое изменение.
5. График релиза. (Две части этого пункта больше не открыты: для
   текущей остановки `HEALING_ACTIVATION_HEIGHT = 22 521` — застрявший
   полный блок — согласно разделу 6.1; релиз с этим RFC — `0.9.0`, и
   `version_compatible` требует пиров `>= 0.9.0`.)

## Приложение B. Разобранный пример

Сам снимок даёт сценарий. Полный блок 22 521 (автор `BD3E…`) добыт
2026-07-05 03:58:50 UTC; выбранные подписывающие блоки 22 522–22 524 —
от якорных ключей `BE46…`, `BBA0…` и `6E24…` — приходят в течение
шести минут; четвёртая выбранная подпись так и не приходит.

По этому RFC (с `HEALING_ACTIVATION_HEIGHT = 22 521`): начиная с
2026-07-10 03:58:50 UTC каждый пустой блок, автор которого — один из
двенадцати ключей выше (минус три уже подписавших и автор полного
блока), а таймштамп не раньше этого момента, является валидной
подписью для блока 22 521. Узлы с этими ключами автоматически ставят
лечащие задачи со случайными задержками. Пусть `9A46…` (недавний пул)
добывает блок 22 525 в 04:11. В окне теперь четыре подписи, три из них
якорные, так что предикат раздела 3.5 выполнен и блок 22 521
зафиксирован. Следующий доменный блок 22 526 разрешён немедленно;
нормальная работа возобновляется.

(Если бы три выбранные подписи пришли от неякорных допущенных ключей,
одна лечащая подпись `9A46…` окно бы *не* зафиксировала — потребовался
бы пятый блок от любого якоря. На практике самые активные выбранные
подписанты и якоря сильно пересекаются, как и здесь.)

Пятидневный разрыв навсегда останется в цепочке честной записью о том,
что произошло лечение; никакой иной след не отличает лечащие блоки от
обычных подписывающих.
