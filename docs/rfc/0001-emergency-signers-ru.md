# RFC-0001: Аварийное восстановление пула подписантов через пороговую подпись

| Поле       | Значение                                            |
|------------|-----------------------------------------------------|
| Номер RFC  | 0001                                                |
| Название   | Аварийное восстановление пула подписантов через пороговую подпись |
| Статус     | Черновик                                            |
| Тип        | Standards Track (изменение консенсуса)              |
| Создан     | 2026-05-05                                          |
| Авторы     | Проект ALFIS                                        |

## Аннотация

В ALFIS используется схема Limited Confidence Proof-of-Activity, в
которой каждый блок с доменом («полный блок») должен быть закрыт
минимум четырьмя подписывающими блоками от кворума из семи
исторически имеющих право подписи ключей. Множество таких
подписывающих ключей строится из авторов прошлых блоков и потому
ограничено ранней историей цепочки. Сегодня это множество
сконцентрировано в примерно шестнадцати ключах, принадлежащих
пяти-шести людям. Цепочка рискует встать навсегда, если критическая
доля этих держателей одновременно станет недоступна.

Этот RFC определяет аварийный механизм восстановления, позволяющий
сообществу разморозить остановившуюся цепочку и заменить пул
подписантов новым, намеренно распределённым списком живых ключей.
Механизм использует пороговую Ed25519-подпись (FROST, RFC 9591),
держателями которой являются `M`-из-`N` участников сообщества.
Аварийный приватный ключ никогда не существует целиком ни на одной
машине ни в один момент времени; он сразу формируется как набор
FROST-долей. Восстановление требует out-of-band социальной
координации плюс одного in-protocol раунда обмена сообщениями через
gossip.

Механизм спроектирован так, чтобы никогда не срабатывать ни в
нормальной работе, ни при кратко- и среднесрочных сетевых сплитах, и
требует явных действий человека даже когда его временное условие
выполнено.

## 1. Мотивация

### 1.1 Текущая модель подписантов

Начиная с индекса блока `BLOCK_SIGNERS_START` (сейчас 35), каждый
полный блок должен быть закрыт как минимум `BLOCK_SIGNERS_MIN`
(сейчас 4) подписывающими блоками, прежде чем будет разрешён
следующий полный блок. Множество из семи имеющих право подписи
ключей для конкретного полного блока выбирается псевдослучайно из
авторов прошлых блоков, фильтруясь по минимальному количеству блоков
`block.index / 100`. См. функцию `get_block_signers()` в
`src/blockchain/chain.rs` для эталонной реализации.

Этот фильтр гарантирует, что свежесозданный ключ атакующего не
сможет дёшево попасть в число подписантов, но имеет побочный эффект:
чем выше растёт цепочка, тем сложнее любому новому ключу когда-либо
пройти этот порог. На момент написания цепочка находится на индексе
~21 735, что требует от ключа ~217 намайненных блоков для прохождения
порога. Множество подписантов фактически заморожено в составе
небольшой группы давних участников.

### 1.2 Bus factor

Независимые наблюдения за множеством имеющих право подписи ключей
показывают, что подбор подписантов фактически контролируется
примерно 16 ключами, принадлежащими 5-6 людям. Если достаточная доля
этих держателей одновременно станет недоступна — из-за смерти,
потери интереса, поломки оборудования, длительного отсутствия или
давления — никакой случайный выбор семи подписантов не даст четырёх
подписей, нужных для продвижения цепочки. Цепочка остановится
навсегда.

Прямое измерение состояния цепочки на высоте 21 755 (2026-05-05)
эмпирически подтверждает эту концентрацию:

* Из 695 уникальных ключей, когда-либо производивших блок, только
  **16** проходят текущий порог `min_block_count = height / 100 =
  217` и могут быть выбраны подписантами.
* Топ-9 из этих 16 ключей производят **88-92%** всех подписывающих
  блоков во всех измеренных окнах (вся цепочка, последние 10 000,
  последние 1 000, последние 100). Концентрация со временем слабо
  растёт, а не уменьшается.
* Из топ-5 самых активных ключей **четыре принадлежат одному
  человеку**. Односторонний уход этого человека одномоментно
  уберёт примерно половину всей текущей подписной активности.

Последнее наблюдение решающее: bus-factor — не далёкая теоретическая
проблема, а реальная операционная.

### 1.3 Положение в координатах CAP

ALFIS сегодня — операционно CP-система: при сетевом сплите обе
половины корректно отказываются производить расходящееся состояние.
Этот RFC сохраняет CP-поведение во всех реалистичных кратко- и
среднесрочных сценариях сплита. Он вводит узкую, требующую участия
человека ветвь к доступности (A), которая активируется только после
интервала, существенно превышающего любой вероятный сплит, и даже
тогда — только при явных криптографических действиях `M` независимых
сторон.

### 1.4 Цели

1. Разрешить разморозку цепочки, когда критическая доля исторических
   подписантов потеряна навсегда.
2. Заменить множество подписантов новым, географически и социально
   разнообразным списком живых ключей с лимитом на человека.
3. Требовать явной социальной координации — механизм не может быть
   запущен одним только автоматическим таймером.
4. Гарантировать, что аварийный приватный ключ никогда не существует
   в восстанавливаемом виде ни на одной машине.
5. Сохранить все существующие правила владения доменами и
   идентичностями: аварийный механизм размораживает цепочку, но не
   даёт никакой возможности захватить чужие домены.
6. Добавить минимально возможную поверхность изменений в консенсус:
   один новый класс блока, одну новую ветвь верификации подписи,
   одно исключение по автору блока для аварийного публичного ключа.

### 1.5 Что НЕ входит в задачи

1. Этот RFC не решает проблему краткосрочной устойчивости к
   цензуре. Враждебный сплит длительностью менее недели по-прежнему
   замораживает цепочку.
2. Этот RFC не вводит универсальную систему управления (governance).
   Аварийный публичный ключ имеет ровно одну функцию: замену
   списка подписантов, когда цепочка остановилась.

## 2. Терминология

| Термин | Определение |
|--------|-------------|
| `EMERGENCY_PUBKEY` | 32-байтный публичный Ed25519-ключ, вшитый в код ALFIS. Соответствующий приватный ключ существует только в виде FROST-долей. |
| Share holder (держатель доли) | Один из `N` участников, владеющих FROST-долей `EMERGENCY_PUBKEY`. |
| Auth-ключ | Отдельная Ed25519-ключевая пара, принадлежащая держателю доли и используемая для аутентификации gossip-сообщений в ходе сессии восстановления. НЕ FROST-доля. |
| Список auth-pubkey | Список публичных auth-ключей текущего набора держателей долей, настраиваемый в конфиге узла как hex-строки. |
| Сессия восстановления | Одна попытка произвести один аварийный блок; идентифицируется хэшем последнего полного блока, который надо разморозить. |
| Аварийный блок | Блок класса `CLASS_SIGNERS_UPDATE`, у которого `pub_key` равен `EMERGENCY_PUBKEY`. Содержит новый список подписантов. |

## 3. Спецификация

### 3.1 Активация

Этот RFC активируется на высоте цепочки
`EMERGENCY_ACTIVATION_HEIGHT` (значение задаётся в момент выпуска
релиза, реализующего этот RFC, и выбирается так, чтобы у узлов было
достаточное время на обновление). Ниже этой высоты правила данного
RFC не действуют.

Узел, не обновлённый до версии с поддержкой этого RFC, отвергнет
аварийные блоки как некорректные. Операторы обязаны обновиться до
достижения `EMERGENCY_ACTIVATION_HEIGHT`.

### 3.2 Аварийный публичный ключ

Один 32-байтный Ed25519-публичный ключ вшит в бинарь как константа
`EMERGENCY_PUBKEY`. Соответствующий приватный ключ создаётся в ходе
церемонии Distributed Key Generation (RFC 9591 раздел 3) среди
исходных `N` держателей долей или, в качестве альтернативы, путём
шамировского разделения с немедленным уничтожением исходного ключа.
См. раздел 7 о долгосрочных аспектах церемонии.

Выбранная схема: `M`-из-`N` = 5-из-9.

### 3.3 Новый класс блока

Вводится новый класс транзакции:

```
CLASS_SIGNERS_UPDATE = "signers_update"
```

Блок этого класса:

1. ОБЯЗАН иметь `pub_key`, равный `EMERGENCY_PUBKEY`.
2. ОБЯЗАН иметь валидную Ed25519-подпись над каноническими байтами
   блока, проверяемую относительно `EMERGENCY_PUBKEY`. Подпись
   создаётся через FROST-агрегацию; с точки зрения верификатора она
   неотличима от обычной Ed25519-подписи.
3. ОБЯЗАН иметь транзакцию, поле `data` которой содержит список из
   7-15 включительно 32-байтных Ed25519-публичных ключей в строго
   возрастающем лексикографическом порядке без дубликатов. Это —
   ключи для замены пула подписантов.
4. ОБЯЗАН иметь `difficulty`, равный `SIGNER_DIFFICULTY` (16).
   Вычислительный барьер существует только для того, чтобы
   привязать блок к цепочке и предотвратить тривиальный блок-спам;
   защитная нагрузка обеспечивается пороговой подписью, а не PoW.

Формат сериализации списка подписантов внутри `data` — это
конкатенация 32-байтных публичных ключей по порядку, опционально
обрамлённая префиксом длины в соответствии с существующей
кодировкой транзакций (точное обрамление делегируется эталонной
реализации, при условии что оно ОБЯЗАНО быть каноническим: для
любого данного списка существует ровно одна валидная кодировка).

### 3.4 Условие активации аварийного блока

Аварийный блок валиден только в том случае, если на момент его
добавления цепочка находится в остановленном состоянии. Конкретно,
пусть `last_full` — это последний полный блок (любой блок, у
которого поле `transaction` непустое, включая предыдущий аварийный
блок). Аварийный блок `B` валиден только если:

```
B.timestamp - last_full.timestamp >= EMERGENCY_TIMEOUT_SECONDS
```

где `EMERGENCY_TIMEOUT_SECONDS = 7 * 86400 = 604800`.

Обоснование: подписывающие блоки в норме приходят в течение минут
после своего полного блока. Промежуток в неделю между полными
блоками означает либо что кворум подписантов не смог собраться,
либо что цепочка намеренно простаивала без новой доменной
активности. Во втором случае операционных оснований выпускать
аварийный блок вообще нет, так что безобидный аварийный блок
эквивалентен no-op обновлению списка подписантов.

Условие проверяется относительно `last_full.timestamp` независимо
от того, появлялись ли промежуточные подписывающие блоки. Полный
блок, закрытый двумя подписями и стоящий неделю, имеет право на
аварийную разморозку точно так же, как полный блок без единой
подписи.

### 3.5 Правило закрытия для аварийного блока

После аварийного блока на высоте `H_e` следующий полный блок (с
доменом) разрешён только после `BLOCK_SIGNERS_MIN` подписывающих
блоков, произведённых подписантами из нового списка, заданного в
`H_e`. Сами подписывающие блоки следуют стандартным правилам; они
НЕ подписываются `EMERGENCY_PUBKEY` и НЕ подпадают ни под какую
ослабленную валидацию. Аварийный механизм производит ровно один
блок за сессию восстановления.

`EMERGENCY_PUBKEY` НЕ ДОЛЖЕН выступать автором никакого
неаварийного блока. Подписывающий блок, авторизованный
`EMERGENCY_PUBKEY`, невалиден. Обычный полный блок, авторизованный
`EMERGENCY_PUBKEY`, невалиден. Аварийный публичный ключ имеет
ровно одну роль: подписывать аварийные блоки класса
`CLASS_SIGNERS_UPDATE`.

### 3.6 Замена пула подписантов

После последнего по высоте аварийного блока на высоте `H_e`
функция `get_block_signers(full_block)` для любого `full_block` с
`full_block.index > H_e` возвращает результат, выбирая из списка
публичных ключей в аварийном блоке `H_e`, А НЕ из исторических
авторов блоков. Механизм выбора сохраняет существующее
сид-производное (`block.signature.get_tail_u64()`), но снимает
фильтр `min_block_count`, поскольку свежие ключи ещё не майнили
блоков.

Аварийный список всегда содержит `K` ключей с 7 ≤ K ≤ 15
(гарантируется на этапе валидации блока, см. раздел 3.3). Параметры
кворума остаются неизменными по сравнению с обычной работой:
`BLOCK_SIGNERS_ALL = 7` кандидатов, `BLOCK_SIGNERS_MIN = 4`
требуется. Выбор 7 кандидатов из K-элементного списка использует
существующий сид-механизм (`tail.wrapping_mul(count) % window`),
применённый к аварийному списку как к массиву кандидатов, со
снятым фильтром `min_block_count` (свежие ключи ещё не майнили
блоков).

Для канонического случая K = 7 все ключи из аварийного списка
выбираются в подписанты, сохраняя исторический кворум 4-из-7. Для
K > 7 детерминистический сид выбирает 7 различных ключей.

Список подписантов с K < 7 невалиден, и аварийный блок, его
несущий, ОБЯЗАН быть отвергнут. См. также раздел 3.3 (инвариант
уровня блока) и раздел 3.7.3 (условие прерывания на уровне сессии).

Последующий аварийный блок на более высокой высоте полностью
заменяет список предыдущего. Аварийные блоки не накапливаются.

### 3.7 Протокол сессии восстановления

Сессия восстановления производит ровно один аварийный блок.
Протокол построен с учётом того, что `block.signature` покрывает
весь блок, включая намайненные поля `nonce` и `random` (см.
`src/blockchain/hash_utils.rs::check_block_signature`). Поэтому
финальная FROST-подпись должна вычисляться только после того, как
найден валидный PoW-нонс, и все держатели долей должны согласиться
на единственный кандидатный блок до начала производства частичных
подписей — round-1 нонсы FROST одноразовые и не могут быть
переиспользованы между несколькими кандидатами.

Протокол состоит из четырёх фаз.

#### 3.7.1 Внеполосная координация (Фаза 0)

Держатели долей координируются через внешний канал (Mimir,
DeltaChat, Matrix, Signal, рассылка и т.п.), чтобы:

1. Подтвердить, что присутствуют и готовы участвовать как минимум
   `M = 5` держателей. Может участвовать и больше пятерых; любое
   число `s` с `5 ≤ s ≤ 9` валидно. Большее `s` повышает
   устойчивость к тому, что один из участников впоследствии не
   доставит свою частичную подпись в Фазе 3, поскольку протокол
   не толерантен к выпадению partial signature после закрытия
   Фазы 1 (см. 3.7.4).
2. Согласовать новый список подписантов. Каждый участвующий
   держатель внесёт от 0 до 3 своих ключей (т.е. ключей, для
   которых он лично владеет соответствующим приватным ключом, или
   делегированных ключей членов сообщества, за которых он ручается).
3. Подтвердить хэш полного блока, который надо разморозить
   (`last_full.hash`); этот хэш служит идентификатором сессии
   восстановления.

Эта фаза целиком социальная и не производит внутрипротокольных
артефактов.

#### 3.7.2 Рассылка commitment (Фаза 1)

Каждый участвующий держатель доли через свой узел рассылает
единственное gossip-сообщение `EmergencyCommitment`:

| Поле | Тип | Описание |
|------|-----|----------|
| `session_id` | 32 байта | Хэш полного блока, который размораживается. |
| `auth_pubkey` | 32 байта | Auth-публичный ключ держателя (должен быть в настроенном списке auth-pubkey). |
| `keys` | список 32-байтных pubkey, длина 0..3 | Ключи, которые этот держатель вносит в новый список подписантов. |
| `frost_commitment` | байты | FROST round-1 hiding+binding commitment по RFC 9591 раздел 5.1. |
| `auth_signature` | 64 байта | Ed25519-подпись от `auth_pubkey` над каноническим кодированием всех предыдущих полей. |

Это сообщение НЕ содержит partial signature: на этом этапе
подписываемого блока ещё не существует, а partial signatures FROST
зависят от подписываемого сообщения, доли подписанта, его нонса и
агрегата commitment'ов всех участников. Ни один из этих входов
ещё не определён полностью.

Принимающие узлы:

1. Проверяют, что `auth_pubkey` присутствует в настроенном списке
   auth-pubkey. Если нет — отбросить.
2. Проверяют `auth_signature` относительно `auth_pubkey`. Если
   невалидна — отбросить.
3. Проверяют, что `keys` содержит максимум 3 элемента без
   дубликатов. Иначе — отбросить.
4. Проверяют, что `session_id` совпадает с хэшем текущего
   `last_full_block`. Если нет — игнорировать.
5. Проверяют, что данный держатель ещё не присылал commitment в
   этой сессии (один commitment на держателя на сессию). Если виден
   дубликат — оставить первый, последующие отбросить.
6. Перенаправляют через стандартный gossip соседям.
7. Кэшируют commitment в памяти до окончания сессии.

Защита от replay-атак: каждое сообщение содержит `session_id =
last_full.hash`. Как только добавлен один аварийный блок, цепочка
продвигается, и хэш последнего полного блока меняется. Сообщения
из предыдущих сессий становятся непригодны для последующих сессий.

#### 3.7.3 Конструирование и майнинг кандидата (Фаза 2)

После получения первого `EmergencyCommitment` для нового
`session_id` каждый узел открывает **окно commitment'ов**
длительностью `T_window = 300 секунд`. В течение окна
дополнительные commitment'ы от различных auth-pubkey принимаются
и распространяются gossip'ом. По закрытии окна множество
участников `P` фиксируется:

* `P` = множество различных auth-pubkey, приславших валидные
  сообщения `EmergencyCommitment` для этого `session_id` до
  закрытия окна.
* Сессия жизнеспособна, если и только если `|P| ≥ M` (т.е.
  закоммитили минимум 5 держателей долей). Если `|P| < M`,
  сессия прерывается; держатели долей могут начать свежую сессию
  с новым раундом внеполосной координации.
* Если `|P| > M`, все `s = |P|` участников связаны с этой
  сессией. FROST-агрегация в Фазе 4 требует partial signatures
  от всех `s` (а не любых `M` из них); порог управляет только
  минимальным количеством участников на входе в Фазу 1, а не
  размером множества участников после.

После закрытия окна каждый узел может детерминированно вычислить:

* Полный финальный список подписантов — взятием объединения всех
  полей `keys` из сообщений в `P`, удалением дубликатов и
  лексикографической сортировкой. Этот список идентичен у каждого
  узла, наблюдающего одинаковое множество commitment'ов.
* Множество FROST-commitment'ов, нужных для вычисления partial
  signatures на следующей фазе.

Если получившийся список ключей содержит меньше 7 различных
ключей, сессия некорректна и ОБЯЗАНА быть прервана; держатели
долей отвечают за обеспечение адекватного распределения вкладов в
Фазе 0. С `s` участниками и потолком 3 ключа на участника
достижимый диапазон — `1..3s`; протоколом разрешено только окно
7..15.

Назначенный **координатор** затем конструирует кандидатный блок и
майнит его. Координатор выбирается детерминированно — это
держатель доли, чей `auth_pubkey` лексикографически наименьший
среди членов `P`.

Координатор конструирует:

* `pub_key = EMERGENCY_PUBKEY`
* `prev_block_hash = last_block.hash`
* `index = last_block.index + 1`
* `timestamp` = текущее wall-clock время
* `difficulty = SIGNER_DIFFICULTY`
* `transaction.class = CLASS_SIGNERS_UPDATE`
* `transaction.data` = каноническое кодирование отсортированного
  без дубликатов списка ключей подписантов
* `transaction.identity`, `transaction.confirmation`,
  `transaction.signing`, `transaction.encryption` = пусто
* `signature` = пусто (placeholder; будет заполнено в Фазе 4)

Затем координатор майнит: ищет `nonce` и `random` такие, что
`hash_difficulty(block.hash) >= SIGNER_DIFFICULTY`, при этом hash
блока вычисляется с занулённым `signature` (согласованно с
`check_block_hash`). Найдя, координатор рассылает кандидата через
новое gossip-сообщение:

| Поле | Тип | Описание |
|------|-----|----------|
| `session_id` | 32 байта | То же, что в Фазе 1. |
| `auth_pubkey` | 32 байта | Auth-pubkey координатора. |
| `candidate_block` | байты | Полностью намайненный блок с занулённым полем `signature`. |
| `auth_signature` | 64 байта | Ed25519-подпись от auth-pubkey координатора над каноническим кодированием предыдущих полей. |

Принимающие узлы проверяют:

1. `auth_pubkey` совпадает с лексикографически наименьшим
   auth-pubkey среди приславших commitment в этой сессии. Если
   нет — отбросить (некоординатор пытается вести сессию).
2. `auth_signature` валидна.
3. Список подписантов в `candidate_block` (декодированный из
   `transaction.data`) точно равен детерминированному объединению
   из commitment'ов Фазы 1.
4. `candidate_block` имеет корректный `pub_key`
   (`EMERGENCY_PUBKEY`), `prev_block_hash` (совпадает с текущим
   `last_block.hash`), `index`, `difficulty`, и hash,
   удовлетворяющий `SIGNER_DIFFICULTY`.
5. Соблюдается условие `timestamp - last_full.timestamp >= 604800`
   (недельное аварийное условие; см. раздел 3.4).

Если все проверки пройдены — gossip'ить сообщение и переходить к
Фазе 3.

**Перехват роли координатора.** Если в течение
`T_coordinator = 600 секунд` после завершения Фазы 1 не наблюдалось
`EmergencyCandidate`, роль переходит к держателю с
лексикографически следующим наименьшим auth_pubkey. Каждый
последующий координатор получает то же 600-секундное окно.
Сорвавшийся координатор мог упасть, быть подвергнутым цензуре или
просто не иметь железа, чтобы намайнить `SIGNER_DIFFICULTY = 16`
за разумное время (на практике — секунды на современном CPU; десять
минут — щедрый запас).

#### 3.7.4 Частичные подписи (Фаза 3)

Каждый держатель доли в `P`, получив и проверив
`EmergencyCandidate`, вычисляет свою FROST round-2 partial
signature над каноническими байтами `candidate_block` (с
занулённым полем `signature` — те же байты, что покрывает
`check_block_signature`). Входы для partial signature:

* Доля подписанта от приватного ключа FROST.
* Round-1 нонс подписанта, сохранённый локально с Фазы 1.
* Агрегат round-1 commitment'ов от всех `s` участников из `P`
  (виден через gossip).
* Байты сообщения (канонические байты блока с занулённой signature).

Держатель рассылает сообщение `EmergencyPartialSignature`:

| Поле | Тип | Описание |
|------|-----|----------|
| `session_id` | 32 байта |  |
| `candidate_hash` | 32 байта | `block.hash` подписываемого кандидата. |
| `auth_pubkey` | 32 байта |  |
| `frost_partial_signature` | байты | Выход FROST round-2. |
| `auth_signature` | 64 байта | Ed25519 над предыдущими полями. |

Держатель доли ОБЯЗАН подписать максимум один кандидат за сессию
(round-1 нонс одноразовый). Если наблюдаются два разных кандидата,
держатели подписывают тот, который провалидировали первым, и
игнорируют второй. На практике правило детерминистического
координатора гарантирует, что в честных условиях существует только
один кандидат на сессию; ограничение «один на сессию» — оборонная
страховка.

**Независимость от порядка.** Partial signatures могут
производиться, рассылаться и агрегироваться в любом порядке.
FROST-агрегация математически коммутативна: финальная подпись —
это сумма всех partial signatures участников (с
лагранжевыми коэффициентами, уже встроенными в каждое `z_i` его
производителем, вычисленными от фиксированного множества `P`).
Любая перестановка partial signatures даёт тот же агрегат.
Конкретно:

* Держатели могут вычислять и рассылать свою partial signature
  сразу после того, как провалидировали кандидата в UI; ждать
  кого-либо не нужно.
* Агрегатор может собирать partial signatures в любую структуру
  данных; сортировать перед суммированием не нужно.
* Повторная доставка уже виденной partial signature через gossip
  безвредна и просто дедуплицируется по
  `(session_id, auth_pubkey)`.

Единственное требование к порядку во всём протоколе — это
**межфазная граница**: каждый участник должен согласиться на
множество `P` (определяется закрытием окна commitment'ов Фазы 2)
и на байты кандидата (определяются Фазой 2) **до** производства
любой partial signature в Фазе 3. После того как эти два факта
зафиксированы, порядок прихода сообщений Фазы 3 неважен.

Принимающие узлы проверяют auth-подпись, проверяют, что
`auth_pubkey` присутствует в `P` (т.е. среди закоммитивших в
Фазе 1 для этой сессии), проверяют, что `candidate_hash`
соответствует активному кандидату, и распространяют сообщение по
gossip.

#### 3.7.5 Агрегация и распространение (Фаза 4)

Любой узел, наблюдающий ровно `s` валидных сообщений
`EmergencyPartialSignature` — по одному от каждого участника `P` —
для одного и того же кандидата, агрегирует их по RFC 9591 раздел
5.3 в полную Ed25519-подпись, помещает её в
`candidate_block.signature` и распространяет получившийся блок
через стандартный путь распространения блоков.

Если в течение разумного окна (например, 5 минут после рассылки
кандидата) пришло меньше `s` partial signatures, сессия мертва:
FROST-агрегация требует partial signatures от **всех**
закоммитивших в Фазе 1, а не от любого подмножества размера `M`.
Порог `M` управляет только минимальным размером `P` на момент
закрытия Фазы 1; после фиксации `P` каждый член `P` обязан
довести до конца Фазу 3, иначе сессия неисправима, и нужно
запускать свежую сессию (с новым раундом координации Фазы 0 и
новыми round-1 commitment'ами).

Затем блок валидируется всеми узлами (включая узлы, не являющиеся
держателями долей) обычной функцией `check_block_signature`
относительно `EMERGENCY_PUBKEY`. Пороговое происхождение подписи
невидимо для верификатора: по построению это обычная
Ed25519-подпись.

### 3.8 Конфигурация: список auth-pubkey

Каждый оператор узла настраивает в `alfis.toml`:

```toml
[emergency]
auth_pubkeys = [
    "<hex-encoded 32-byte Ed25519 pubkey of share holder 1>",
    "<hex-encoded 32-byte Ed25519 pubkey of share holder 2>",
    # ... up to share holder N
]
```

Список содержит auth-публичные ключи всех `N = 9` текущих
держателей долей. Этот список обновляется в координации с
церемониями ресэширинга (раздел 7).

Сам `EMERGENCY_PUBKEY` остаётся compile-time константой.
Конфигурируемы только auth-ключи, используемые для фильтрации
gossip-сообщений.

### 3.9 Программное обеспечение узла держателя доли

Держатель доли дополнительно:

1. Загружает свою FROST-долю из файла `emergency-key.part`,
   расположенного рядом с обычным `.key`-файлом. Формат
   делегируется эталонной реализации; концептуально файл несёт
   одну FROST-долю плюс групповой публичный ключ для проверки на
   вменяемость.
2. Запускает дополнительный экран UI (фича web GUI) под
   названием «Emergency coordination» с элементами управления:
   * Загрузить или выгрузить файл доли в runtime.
   * Просмотреть состояние ожидающей сессии восстановления (какие
     commitment'ы Фазы 1 видны, виден ли кандидат Фазы 2, какие
     partial signatures Фазы 3 видны).
   * Ввести предлагаемый список ключей (до 3-х), которые этот
     держатель хочет внести, и инициировать рассылку
     `EmergencyCommitment` (Фаза 1).
   * Если этот держатель — детерминистический координатор сессии:
     инициировать конструирование и майнинг кандидата, затем
     рассылку `EmergencyCandidate` (Фаза 2).
   * При появлении валидного кандидата: инициировать создание и
     рассылку своей `EmergencyPartialSignature` (Фаза 3).
3. НЕ участвует автоматически. Каждое протокольное сообщение
   требует явного действия оператора через UI. Держатель доли
   ОБЯЗАН осознанно одобрить сессию, предложенный список
   подписантов и кандидатный блок до того, как будет произведена
   любая partial signature.

Узел, не являющийся держателем доли, не имеет специального UI и
дополнительных файлов; он просто распространяет аварийные
сообщения через gossip и валидирует блоки как любой другой узел.

## 4. Обоснование

### 4.1 Почему FROST, а не реконструкция по Шамиру

Наивный дизайн вёз бы шамировские доли приватного ключа
`EMERGENCY_PUBKEY`, передавал бы доли от `M` держателей одному
выбранному узлу, который реконструировал бы ключ в памяти,
подписывал, а затем «забывал». Этот дизайн ломается тремя
способами:

1. Реконструирующий узел кратковременно держит полный приватный
   ключ. С этого момента он криптографически неотличим от узла,
   сохранившего копию. Аварийная привилегия становится
   единоличной — на усмотрение того, кого выбрали реконструировать.
2. Атаки раскрытия памяти (RAM-дампы, swap, гипервизорный
   интроспекшн, malware) на реконструирующий узел компрометируют
   ключ навсегда.
3. Последующие аварийные блоки становятся неотличимы от
   единоличного действия предыдущего реконструктора.

FROST исключает все три проблемы. Приватный ключ никогда не
существует как восстанавливаемый объект. Каждый держатель доли
производит partial signature, которая математически фиксирует его
обязательство для конкретного сообщения; partial signatures нельзя
скомбинировать в приватный ключ, можно только в подпись для уже
согласованного сообщения. Следующая аварийная ситуация требует
такой же кооперации.

### 4.2 Почему таймаут — неделя

Подписывающие блоки в норме приходят в течение минут. Задержка в
часы — уже аномалия, но правдоподобная (сетевой сплит, краткий
сбой). Задержка в неделю неправдоподобна ни в одном реалистичном
операционном сценарии, кроме отказа кворума. Выбор недели
балансирует:

* У владельцев доменов, истекающих в ближайшее время (в пределах
  `DOMAIN_RENEW_TIME` = 30 дней), остаётся осмысленное время для
  обновления после восстановления.
* Краткосрочные сплиты (часы, дни) не могут случайно вызвать
  расходящиеся аварийные блоки на каждой стороне, потому что ни
  одна сторона не простаивала достаточно долго.
* Восстановление достаточно быстрое, чтобы быть операционно
  значимым, а не многомесячным процессом.

### 4.3 Почему список auth-ключей в конфигурации, а не в консенсусе

Если бы auth-ключи были частью консенсуса, любая ротация
держателя доли требовала бы хардфорка или специального класса
транзакций. Хранение их в конфигурации узла перекладывает
ответственность на операторов: auth-ключи — это фильтр на
gossip-уровне, а не на уровне валидности блока. Неправильно
сконфигурированный узел просто не сможет распространить аварийные
сообщения, но всё равно корректно провалидирует получившийся
аварийный блок (поскольку подпись блока проверяется напрямую
относительно `EMERGENCY_PUBKEY`).

### 4.4 Почему максимум 3 ключа на держателя

Если один держатель доли вносит больше 3 ключей, он существенно
формирует получающийся пул подписантов, частично перечёркивая
цель распределения. Тройка — компромисс: достаточно, чтобы
держатель мог поручиться за двух членов сообщества плюс себя, и
мало настолько, чтобы ни один держатель не доминировал. Лимит
обеспечивается социально через gossip-уровневую валидацию: каждое
сообщение несёт auth-подпись держателя над полем `keys`, поэтому
злонамеренный держатель не может подставить другого. Держатель,
превысивший лимит, отбрасывается на gossip-уровне всеми честными
пирами.

### 4.5 Почему difficulty 16 (SIGNER_DIFFICULTY)

Более высокая PoW (24 или 28) рассматривалась, но не даёт
дополнительной защиты: атакующего, скомпрометировавшего пять
FROST-долей, не остановят несколько лишних секунд хэширования.
Сама пороговая подпись и есть защитный барьер. Более низкая PoW
была бы приемлема, но существующая константа `SIGNER_DIFFICULTY`
уже откалибрована для быстрого производства блоков обычными узлами
и переиспользует существующее поведение майнинг-цикла.

### 4.6 Почему `M` — минимум, а не точное число участников

`M = 5` — это FROST-порог: минимальное число держателей долей,
требуемое чтобы вообще произвести подпись. Это НЕ требуемый размер
множества участников в конкретной сессии. Любое число `s` с
`5 ≤ s ≤ 9` — валидный размер сессии, и FROST-агрегация корректно
производит одну Ed25519-подпись для любого такого `s`.

На практике держатели долей должны согласовать в Фазе 0 множество
участников, которым они коллективно доверяют довести дело до
конца, и придерживаться этого множества. Возможность выбрать
`s > 5` предоставлена для того, чтобы аварийная процедура
масштабировалась до того подмножества держателей, которое
оказалось доступно, не требуя ровно пятерых.

## 5. Соображения безопасности

### 5.1 Компрометация списка auth-pubkey

Если атакующий сможет подменить настроенный список auth-pubkey на
узле жертвы, такой узел будет принимать злонамеренные сообщения и
отвергать честные. Однако получившийся аварийный блок всё равно
обязан нести валидную FROST-агрегатную подпись, что требует
кооперации `M` настоящих держателей долей. Атакующий, таким
образом, получает возможность цензурировать восстановление на
изолированных узлах, но не может подделать восстановление.
Операторы ОБЯЗАНЫ относиться к настройке `auth_pubkeys` с той же
аккуратностью, что и к origin или DNS-апстриму.

### 5.2 Компрометация `M` FROST-долей

Если атакующий получит 5 из 9 FROST-долей одновременно, он сможет
произвести валидный аварийный блок в любой момент, когда
выполнено условие таймаута. Он НЕ МОЖЕТ использовать его, чтобы
захватить существующие домены (раздел 3.5 — все стандартные
правила доступности identity продолжают действовать). Он МОЖЕТ
заменить список подписантов на подконтрольный ему. С этого момента
обычное производство блоков продолжается под его подписантами,
которые могут майнить новые (неконфликтующие) домены на стандартных
скоростях и подписывать блоки друг друга.

Меры смягчения:
* Географическое, юрисдикционное и социальное разнообразие при
  подборе держателей долей.
* Периодический ресэширинг (раздел 7) для аннулирования любых
  тихо утёкших долей.

### 5.3 Replay-атаки между сессиями

Сообщения привязаны к `session_id = last_full.hash`. После любого
успешного аварийного блока цепочка продвигается, и хэш последнего
полного блока меняется (на сам аварийный блок). Старые сообщения
становятся бесполезны. Риска переноса сообщений предыдущей сессии
в новую нет.

### 5.4 Отказ в восстановлении через флуд gossip

Атакующий без легитимных auth-ключей может попытаться залить
gossip-слой фальшивыми сообщениями, чтобы затмить настоящие.
Фильтр auth-ключей в шаге 1 раздела 3.7.2 отбрасывает их сразу же
на каждом узле, так что стоимость ограничена обычными
gossip-механизмами rate-limit. Дополнительные меры не требуются.

### 5.5 Случайное восстановление во время сплита

Сетевой сплит длительностью меньше недели не может вызвать
восстановление ни на одной стороне, потому что таймаут раздела
3.4 не выполнен. Сплит длительностью больше недели трактуется как
эквивалент отказа кворума, и восстановление разрешено. Если сплит
заживёт после восстановления, действуют стандартные правила выбора
цепочки: сторона, произведшая аварийный блок, продвинулась дальше
и побеждает. Сплит становится хардфорком в операционном смысле;
это сознательная цена выбора availability над consistency в
крайнем хвосте.

Этот RFC занимает позицию, что сплит длительностью больше недели
операционно неотличим от смерти кворума и должен трактоваться так
же.

### 5.6 Потеря более `N - M` долей

Если 5 или больше из 9 держателей навсегда потеряют свои доли, ни
разу не поучаствовав в ресэшеринге, аварийный блок никогда не
сможет быть произведён. Это деградирует к до-RFC сценарию отказа и
требует хардфорка для восстановления. Раздел 7 описывает
операционную дисциплину, призванную это предотвращать.

## 6. Обратная совместимость

Этот RFC вводит новый класс транзакции и новую ветвь валидации
блоков. Узлы, которые не были обновлены, отвергнут аварийные блоки
как некорректные и перестанут принимать последующие блоки,
строящиеся на них. Все операторы ОБЯЗАНЫ обновиться до
`EMERGENCY_ACTIVATION_HEIGHT`.

Аварийный механизм не затрагивает ни один блок до активации.
Валидация исторической цепочки не меняется.

Узел, который обновился, но не получает аварийный блок, не
испытывает никаких изменений в поведении. Новые ветви кода спят,
пока не наблюдается аварийный блок.

## 7. Долгосрочное управление ключами

Множество FROST-долей и список auth-pubkey будут эволюционировать
на протяжении жизни проекта. Этот RFC не нормативно специфицирует
протокол ресэширинга; это отнесено к последующему RFC. Здесь
обозначен предполагаемый операционный шаблон:

* **Каденс**: минимум раз в календарный год, желательно чаще в
  первые годы развёртывания.
* **Участники**: минимум `M = 5` текущих держателей долей плюс
  любые входящие или исходящие держатели.
* **Процедура**: проактивный ресэширинг FROST по RFC 9591
  раздел 6, производящий новые доли для того же
  `EMERGENCY_PUBKEY`. После церемонии все старые доли становятся
  бесполезны. Новые auth-pubkey публикуются; операторы узлов
  обновляют `alfis.toml`.
* **Публичный сигнал**: каждый ресэширинг анонсируется публично
  с указанием identity новых держателей долей (но не самих долей).
* **Эскалация**: если ресэшеринга не было два года, сообщество
  должно расценить это как мягкую тревогу и приоритизировать
  организацию церемонии. Если 5 или более долей считаются
  потерянными или скомпрометированными до того, как удалось
  организовать ресэширинг, сообществу СЛЕДУЕТ трактовать это как
  ситуацию для хардфорка и готовить новый RFC.

## 8. Эталонная реализация

Будет предоставлена в отдельном коммите. Реализация должна:

1. Добавить `frost-ed25519` (реализация RFC 9591 от Zcash
   Foundation) как зависимость.
2. Ввести константу `CLASS_SIGNERS_UPDATE` в
   `src/commons/constants.rs`.
3. Расширить валидацию `Transaction` и `Block` в
   `src/blockchain/chain.rs` для распознавания нового класса и
   условия активации.
4. Изменить `get_block_signers` в `src/blockchain/chain.rs`, чтобы
   она консультировалась с последним аварийным блоком, прежде чем
   откатываться на исторический выбор по авторам.
5. Добавить три новых типа gossip-сообщений в
   `src/p2p/message.rs`: `EmergencyCommitment` (Фаза 1),
   `EmergencyCandidate` (Фаза 2), `EmergencyPartialSignature`
   (Фаза 3), с соответствующей логикой распространения,
   дедупликации и состояния сессии в `src/p2p/network.rs`.
6. Добавить секцию `[emergency]` в парсинг настроек в
   `src/settings.rs`.
7. Добавить панель «Emergency coordination» в web UI
   (`src/webview/`) под существующим feature-flag GUI.
8. Добавить загрузчик `emergency-key.part` по аналогии с загрузкой
   keystore.

## 9. Открытые вопросы

1. Точное значение `EMERGENCY_ACTIVATION_HEIGHT`. Будет задано в
   момент релиза на основании наблюдаемой высоты цепочки плюс
   буфер на обновление (например, height + 2000).
2. Каноническое кодирование списка подписантов внутри
   `transaction.data`. Префикс длины vs конкатенация фиксированной
   ширины. Тривиально решается на этапе реализации.
3. Выводить ли состояние аварийной сессии в стандартный UI, или
   только в панель UI для держателей долей. Аргумент за: любой
   оператор узла сможет мониторить ход восстановления. Аргумент
   против: лишняя поверхность.

## Приложение A. Развёрнутый числовой пример

Допустим:

* Высота цепочки на момент активации: 21 735.
* `EMERGENCY_ACTIVATION_HEIGHT = 23 735` (height + 2000).
* Все 9 исходных держателей долей завершили DKG до активации.

После активации цепочка достигает блока 25 000 (полный блок,
последний полный — 25 000) и встаёт. Подписывающие блоки 25 001 и
25 002 приходят, после чего блоки не появляются неделю.

В момент `block 25000.timestamp + 604800` пять держателей долей —
Алиса, Боб, Кэрол, Дэйв и Ева — координируются через Mimir.
Остальные четыре держателя недоступны (один в путешествии, у
другого сломалось железо, двое просто оффлайн). Пятеро — ровно
порог; множество участников `s = 5`. Они согласовывают следующие
вклады:

* Алиса: 3 ключа (свой плюс 2 члена сообщества, за которых
  ручается).
* Боб: 3 ключа.
* Кэрол: 2 ключа.
* Дэйв: 2 ключа.
* Ева: 1 ключ.

Итого: 11 различных ключей.

(В более комфортном сценарии участвовали бы шесть-семь держателей;
протокол работает идентично с `s ∈ {5..9}`. Польза от `s > 5`
чисто социальная — возможность over-recruit'ить заведомо надёжных
участников — поскольку FROST-агрегация требует partial signatures
от **всех** `s` участников, а не от подмножества.)

**Фаза 1 (commitment'ы).** Каждый из пяти держателей рассылает
`EmergencyCommitment` со своим полем `keys` и своим FROST round-1
commitment. Каждый commitment подписан auth-ключом держателя и
распространяется через gossip. Через 30 секунд все узлы
держателей (и все слушающие узлы) видели все пять commitment'ов и
могут вычислить детерминированное объединение ключей (11 ключей,
лексикографически отсортированных).

**Фаза 2 (конструирование и майнинг кандидата).** Допустим, у
Алисы лексикографически наименьший auth-pubkey среди пятерых.
Тогда Алиса — детерминистический координатор. Её узел конструирует
кандидатный блок (класс `CLASS_SIGNERS_UPDATE`,
`pub_key = EMERGENCY_PUBKEY`,
`prev_block_hash = block_25002.hash`, `index = 25003`,
`transaction.data` = каноническое кодирование списка из 11
ключей, `signature` = пустой placeholder). Алиса майнит: при
`SIGNER_DIFFICULTY = 16` это обычно занимает существенно меньше
минуты на современном CPU. Алиса рассылает получившееся
сообщение `EmergencyCandidate`. Узлы остальных держателей долей
получают и валидируют кандидата.

**Фаза 3 (partial signatures).** Каждый из Боба, Кэрол, Дэйва и
Евы (и сама Алиса) одобряет кандидата через UI и инициирует
создание своей FROST round-2 partial signature, вычисленной над
каноническими байтами кандидата (с занулённым полем `signature`)
и агрегатом всех пяти round-1 commitment'ов. Каждый рассылает
`EmergencyPartialSignature`.

**Фаза 4 (агрегация).** Первый узел, который увидел все пять
partial signatures, агрегирует их в одну Ed25519-подпись,
вставляет её в поле `signature` кандидата и рассылает получившийся
блок через стандартный путь распространения блоков. С этого
момента блок — это просто обычный подписанный блок; действуют
обычные валидация и распространение.

Блок 25 003 принимается всеми обновлёнными узлами. Начиная с блока
25 004 возобновляется стандартное производство: следующий доменный
майнер производит полный блок на 25 004; подписанты из нового
списка из 11 ключей (выбранные до 7 детерминистическим
сид-механизмом, без фильтра `min_block_count`) производят
подписывающие блоки 25 005-25 008; цепочка возобновляется.

Аварийный механизм спит до тех пор, пока не произойдёт следующий
недельный простой.
