# RFC-0003: Необратимое исключение мёртвых ключей подписантов

| Поле       | Значение                                              |
|------------|-------------------------------------------------------|
| Номер RFC  | 0003                                                  |
| Название   | Необратимое исключение мёртвых ключей подписантов     |
| Статус     | Черновик                                              |
| Тип        | Standards Track (изменение консенсуса)                |
| Создан     | 2026-07-14                                            |
| Авторы     | Проект ALFIS                                          |
| Связан с   | RFC-0002 (Самовосстановление цепочки), RFC-0001       |

## Аннотация

RFC-0002 позволяет остановившемуся окну фиксации завершиться после
таймаута лечащими подписями из резервного пула. Он намеренно не менял
выбор подписантов, и последствие теперь измерено на живой цепочке:
мёртвые ключи продолжают выбираться для каждого нового полного блока,
поэтому почти каждое окно останавливается и лечится, ограничивая
цепочку примерно одним полным блоком за лечащий таймаут — бессрочно.

Этот RFC добавляет недостающую обратную связь. Каждое вылеченное окно
фиксации — это записанное в цепочке **доказательство отсутствия**:
выбранные подписанты, не создавшие ничего за время остановки
длительностью не менее `HEALING_TIMEOUT` дней, на глазах у всего
резервного пула, считаются мёртвыми. Такие ключи **исключаются
навсегда**: выбор подписантов и ранжирования резервного пула всех
последующих полных блоков их пропускают. Множество исключённых —
чистая функция данных цепочки: ни новых сообщений, ни новых полей
блока, ни конфигурации.

Механизм ограничен двумя предохранителями: **пол кристаллизации**
(исключения не применяются, если они сократили бы допущенную популяцию
ниже `BAN_POOL_FLOOR` ключей) и **минимум на момент выбора** (выбор,
которому не хватает различных ключей, детерминированно временно
возвращает последних исключённых вместо вечного цикла — закрывая уже
существующую опасность незавершаемости в `get_block_signers`, которую
растущий порог допуска мог вызвать и без всяких исключений).

## 1. Мотивация

### 1.1 Измеренное поведение после внедрения лечения

RFC-0002 активировался на высоте 22 521 (остановка 2026-07-05).
Снимок на высоте 22 529 (2026-07-11):

* Застрявшее окно полного блока 22 521 вылечено после 2026-07-10 и
  зафиксировано 2026-07-11 — шесть дней на один полный блок.
* Следующее окно (полный блок 22 526) набрало три выбранные подписи за
  четыре минуты и остановилось: четвёртая выбранная подпись снова
  принадлежит ключу, которого как работающего узла больше не
  существует. Это окно, как ожидается, вылечится через
  `HEALING_TIMEOUT_FAST` (три дня), и следующее за ним, и так далее.
* Из 16 ключей, проходящих порог допуска подписантов
  (`height/100 ≈ 225` созданных блоков), последние созданные блоки
  датируются:

  | Последний созданный блок | Ключи |
  |--------------------------|-------|
  | 2026-07-11 (живы)        | `A980…`, `9B89…`, `BBA0…`, `6E24…` |
  | 2026-07-05 (жив)         | `BE46…` |
  | 2026-07-03/04            | `2A02…`, `BBE5…`, `268E…`, `BC10…` |
  | 2026-06-26               | `4116…`, `9A46…`, `1502…` |
  | 2026-06-16               | `B830…`, `CEE7…` |
  | 2026-05-28               | `22DF…` |
  | 2025-09-14               | `B7FB…` |

Выбор для каждого полного блока отбирает 7 из 15 допущенных ключей
(без автора), взвешенных по числу созданных блоков за всё время.
Четыре доказанно мёртвых ключа (`4116…`, `1502…`, `9A46…`, `B7FB…` —
в точности неявившиеся вылеченного окна блока 22 521) несут 5 064 из
18 722 блоков суммарного веса допущенных (~27%). При такой доле
мёртвого веса в урне ожидаемое число мёртвых ключей на выбор из семи —
около 3, и фиксация 4-из-7 срывается примерно на каждом втором окне.
Цепочка не выздоравливает между лечениями, потому что **ничто никогда
не убирает мёртвый ключ из выбора**: ранжирование за всё время — это
односторонний накопитель, и ключ, мёртвый с 2025 года, всё ещё будет
выбираться на высоте 200 000.

### 1.2 Лечение производит ровно то свидетельство, которое нужно

Выводить смерть из одного лишь молчания некорректно: ключ, который
просто не был выбран, не имел обязанности подписывать, а ключ,
выбранный для окна, зафиксировавшегося за четыре минуты, не имел и
осмысленного шанса. Но *вылеченное* окно — качественно иное:

* у выбранных подписантов была обязанность (их выбрали);
* у них было время (не менее `HEALING_TIMEOUT` — пять дней — публично
  видимой остановки, или три дня в деградировавшем режиме);
* у них было недвусмысленное знание (их собственный узел вычисляет
  обязанность подписи автоматически; подпись не требует действий
  человека);
* и исход записан навсегда (последовательность блоков вылеченного
  окна).

Выбранный ключ, не создавший ничего за такое окно, мёртв во всех
практических смыслах, и свидетель — сама цепочка. Этот RFC превращает
свидетельство в консенсус: доказательство отсутствия, по одному
вылеченному окну за раз.

### 1.3 Почему исключение, а не уже отвергнутые конструкции

RFC-0002, раздел 1.3, отверг **скользящее окно допуска** (атакующий
дёшево намайнивает блоков, становится «недавно активным», входит в
выбор и исчезает или становится враждебным). Этот RFC его не
возвращает: допуск по-прежнему *зарабатывается* как раньше (медленным
накоплением `height/100`), и ни одно правило здесь никого не
принимает. Исключение — противоположная полярность: **негативное
свидетельство только удаляет**. Оно даже напрямую наказывает атаку
«намайнил и исчез»: вгрызшийся ключ, замолчавший будучи выбранным,
исключается после первого же вылеченного окна вместо того, чтобы
вечно отравлять выборы. И его нельзя нацелить на других: единственный
ключ, который атакующий может заставить молчать, — его собственный
(об оговорке с eclipse-атакой см. 5.1).

Также этот RFC сохраняет решение RFC-0002 о том, что всё состояние
выводится из данных цепочки: множество исключённых не требует ни
конфигурации, ни gossip-обмена, ни церемонии.

## 2. Терминология

Термины RFC-0002 (полный блок, окно фиксации, выбранные подписанты,
якорный пул `A(F)`, недавний пул `R(F)`, резервный пул `P(F)`,
лечащая подпись, вылеченное окно) используются без изменений. Новые
термины:

| Термин | Определение |
|--------|-------------|
| Закрывающий блок `F'` | Полный блок, завершающий окно фиксации полного блока `F` (следующий полный блок после `F`). |
| Множество неявившихся `N(F)` | Выбранные подписанты вылеченного окна `F`, не создавшие ни одного блока с индексом в `(F.index, F'.index]`. |
| Множество допущенных `E(H)` | Ключи, проходящие существующий фильтр подписантов на высоте `H`: число созданных блоков на `H` ≥ `minimum_block_count(H)` (`H/100`). |
| Множество исключённых `B(H)` | Все ключи, исключённые кристаллизациями на полных блоках с индексом ≤ `H` (раздел 3.3). |
| Кристаллизация | Детерминированный акт добавления неявившихся окна в множество исключённых, вычисляемый на закрывающем блоке этого окна. |

## 3. Спецификация

### 3.1 Константы

```
BAN_ACTIVATION_HEIGHT  = 22 529   (вершина цепочки на момент релиза, см. 6)
BAN_POOL_FLOOR         = 10       (кристаллизация останавливается на этой популяции)
BAN_DRAW_MIN           = 8        (выбор никогда не видит меньше допущенных ключей)
```

Исключения влияют только на выбор подписантов и вывод резервного пула
для полных блоков с `index > BAN_ACTIVATION_HEIGHT`. Историческая
валидация не меняется.

### 3.2 Вывод неявившихся

Для каждого полного блока `F'` с
`F'.index > HEALING_ACTIVATION_HEIGHT`, закрывающего окно фиксации
предыдущего полного блока `F`:

1. Определить, является ли окно `F` **вылеченным**, структурным тестом
   RFC-0002 (в нём есть пустой блок, автор которого не входит в
   выбранных подписантов `F`).
2. Если вылечено — вычислить множество неявившихся:

   ```
   N(F) = drawn(F) \ { авторы всех блоков с индексом в (F.index, F'.index] }
   ```

Множество авторов намеренно включает сам `F'`: выбранный подписант,
который не подписал, но добыл закрывающий полный блок, доказал, что
жив, и неявившимся не считается. Выбранный подписант, подписавший
*поздно* — после появления лечащих подписей, даже после фиксации, —
также засчитывается: отсутствие оценивается по всему окну, а не по
какому-либо его отрезку. Окна, зафиксировавшиеся без лечения, не
кристаллизуют ничего: их молчавшие выбранные ключи просто не
понадобились, и их молчание ничего не доказывает (здоровое окно
закрывается за минуты).

Каждая величина выше — чистая функция блоков с индексом `≤ F'.index`,
поэтому все узлы, включая синхронизирующиеся с генезиса, выводят
идентичные множества неявившихся.

### 3.3 Кристаллизация и пол

Множество исключённых строится обработкой закрывающих блоков в порядке
цепочки. На каждом закрывающем блоке `F'` с непустым `N(F)`:

1. Упорядочить кандидатов из `N(F)` по **убыванию** числа созданных
   блоков на момент `F'.index`; при равенстве — по возрастанию
   лексикографического порядка байтов публичного ключа. (Убывание:
   самый тяжёлый мёртвый ключ отравляет больше всего выборов, поэтому
   когда пол ограничивает число применимых исключений, первыми
   удаляются самые вредные.)
2. Для каждого кандидата в этом порядке: добавить его в множество
   исключённых **тогда и только тогда**, если после этого
   `|E(F'.index) \ B| ≥ BAN_POOL_FLOOR`. Кандидат, нарушающий пол,
   **пропускается, а не откладывается** — он остаётся допущенным и не
   несёт никакого состояния. Если он действительно мёртв, он снова
   будет выбран, снова не явится и кристаллизуется на более позднем
   закрывающем блоке, когда пол позволит (например, после того как
   другие ключи перерастут порог допуска).

Исключения **необратимы**: никакой блок, таймаут или активность не
удаляет ключ из `B`. (Обоснование в 4.3; предохранитель из 3.4
временно возвращает ключи в отдельные выборы, но никогда не снимает
исключение.)

### 3.4 Влияние на выбор подписантов

Для полного блока `G` с `G.index > BAN_ACTIVATION_HEIGHT`
псевдослучайный обход `get_block_signers(G)` пропускает авторов из
`B(G)` точно так же, как пропускает авторов ниже порога
`minimum_block_count`. `B(G)` включает кристаллизации на закрывающих
блоках с индексом `≤ G.index` — включая кристаллизацию на самом `G`,
если `G` закрывает вылеченное окно; цикличности нет, поскольку
неявившиеся того окна полностью определяются блоками,
предшествующими `G`.

**Минимум на момент выбора (гарантия завершаемости).** Если
`|E(G.index) \ B(G)| < BAN_DRAW_MIN`, исключённые ключи временно
возвращаются *только для этого выбора*, в обратном порядке
кристаллизации (последние исключённые — первыми; внутри одной
кристаллизации — в обратном порядке пункта 3.3), пока счёт не
достигнет `BAN_DRAW_MIN`. Это ограничивает обход: при не менее чем 8
допущенных ключах 7 различных подписантов без автора существуют
всегда. Заметьте, что этот предохранитель нужен и независимо от
исключений: порог `height/100` растёт на единицу каждые 100 блоков и
сокращает `E(H)` сам по себе, а цикл
`while set.len() < BLOCK_SIGNERS_ALL` в текущей реализации не имеет
другого выхода. Временный возврат — вычисление на один выбор, а не
снятие исключения: ключ остаётся в `B` и остаётся исключённым из
окон, где минимум соблюдён.

### 3.5 Влияние на резервный пул

Для полного блока `F` с `F.index > BAN_ACTIVATION_HEIGHT`
ранжирования RFC-0002 пропускают исключённые ключи:

* `A(F)` — топ `HEALING_POOL_ANCHORS` по числу блоков за всё время
  **среди ключей не из `B(F.index)`**;
* `R(F)` — топ `HEALING_POOL_RECENT` недавнего окна, исключая `A(F)`
  **и `B(F.index)`**.

Следующие ключи каждого ранжирования поднимаются. Якорный пул тем
самым отслеживает живых: доказанно мёртвые якоря перестают
обесценивать требование `HEALING_ANCHORS_MIN`, а якоря-заместители
по-прежнему выбираются неподделываемым ранжированием за всё время
(см. 4.5). Минимум из 3.4 здесь не применяется: маленький резервный
пул безопасен (он лишь уменьшает лечащую ёмкость), тогда как
голодающий обход выбора не завершается.

### 3.6 Чего исключение не делает

* **Права на домены не затронуты.** Исключённый ключ может
  регистрировать, продлевать и передавать домены как раньше; его
  полные блоки валидны, и счётчик его созданных блоков продолжает
  расти (счётчик просто больше не влияет на отбор).
* **Открытые окна сохраняют состав.** Выбранный набор и резервный пул
  полного блока фиксируются при его добыче; кристаллизовавшееся позже
  исключение не делает недействительной подпись в уже открытом окне,
  и ключ, выбранный до своего исключения, всё ещё может подписать то
  окно.
* **Никакого пересмотра истории.** Блоки на высоте
  `BAN_ACTIVATION_HEIGHT` и ниже валидируются в точности как раньше.
* **Никаких изменений протокола обмена.** Множество исключённых
  каждый узел вычисляет локально; не добавляется ни сообщение, ни
  поле, ни класс транзакции.

### 3.7 Операционная видимость (вне консенсуса, обязательное поведение)

Узел ОБЯЗАН записывать в лог на уровне `warn` — и отображать как
событие UI там, где UI есть, — каждую выполненную им кристаллизацию:
исключённые ключи, закрывающий блок и вылеченное окно, доказавшее
отсутствие. Исключение ключа — событие уровня управления сетью; оно
никогда не должно происходить беззвучно.

## 4. Обоснование

### 4.1 Почему только негативное свидетельство

Любое правило, *принимающее* ключи по недавнему поведению,
подделываемо намайниванием (RFC-0002, раздел 1.3, пункт 2). Правило,
которое лишь *удаляет* ключи за доказанное неисполнение обязанности,
не принимает никого, и его нельзя фармить. Худшее, что атакующий может
сделать с механизмом исключения, — добиться исключения собственных
ключей. Эта асимметрия и есть суть.

### 4.2 Почему исключаться могут только выбранные подписанты

Только у выбранных подписантов окна есть обязанность его подписать.
Члены резервного пула, не полечившие, не доказывают ничего: лечение —
добровольная избыточная ёмкость, несколько ключей пула соревнуются за
считанные слоты, и самые быстрые вытесняют остальных. Мёртвые *якоря*
этим RFC всё равно удаляются — просто через их роль выбранных
подписантов: топ-ключи за всё время доминируют в весе выбора,
выбираются постоянно и потому, умерев, быстро не являются — после чего
пункт 3.5 убирает их и из якорного ранжирования.

### 4.3 Почему исключения необратимы

Альтернативы были рассмотрены и отвергнуты:

* **Возврат добычей полного блока.** Воскрешение через proof-of-work
  звучит справедливо, но продления доменов освобождены от 24-часового
  кулдауна и получают скидку (известный пробел из RFC-0002, раздел
  5.1), поэтому враждебный ключ мог бы циклить: исчез → исключён →
  одно дешёвое продление → возвращён → выбран → исчез, — взимая с сети
  многодневную заморозку за цикл.
* **Истечение через N блоков.** Детерминировано, но каждое истечение
  действительно мёртвого ключа заново вставляет его в выбор и в итоге
  стоит ещё одного вылеченного окна для повторного исключения —
  периодические самопричинённые остановки навсегда.
* **Растущая цена возврата.** Предотвращает цикл, но добавляет
  счётчики нарушений в состояние консенсуса ради краевой выгоды.

Необратимость — простейшее правило, никогда не возвращающее труп.
Цена его ложных срабатываний реальна, но ограничена: триггер требует,
чтобы *конкретный* ключ промолчал сквозь пятидневную публично видимую
остановку всей цепочки (три дня — только в уже деградировавшем
режиме), что далеко за пределами любого добросовестного сбоя с
перезапуском, а ошибочно изгнанный оператор сохраняет все права на
домены (3.6). Он теряет роль подписанта; сеть теряет одного
подписанта, без которого уже показала, что может жить (она вылечилась
без него). Реальность «ключи ≠ люди» режет и здесь: цепочка в принципе
может исключить только ключ, а вернувшийся оператор — человек. Его
путь назад к подписанию — это проблема входа в допуск, которая вне
рамок этого RFC, но явно обозначена в разделе 7.

### 4.4 Почему отсутствие оценивается по всему окну

Оценка «до первой лечащей подписи» исключила бы выбранный ключ,
подписавший на шестой день, — живой, полезный и наказанный за
медлительность. Оценка на момент фиксации упустила бы автора
закрывающего блока. Правило по всему окну (`(F.index, F'.index]`)
учитывает каждое возможное доказательство жизни, которое цепочка
способна записать, поэтому исключение означает ровно: *этот ключ не
создал ничего, любого рода, за всё остановленное окно вместе с
последствиями лечения*.

### 4.5 Почему фильтруется и резервный пул

Два из семи текущих якорей входят в первую партию исключаемых. Якоря
существуют, чтобы контролировать лечение (`HEALING_ANCHORS_MIN`);
мёртвые якоря — мёртвый груз ровно в том механизме, который
компенсирует мёртвых подписантов. Фильтрация безопасна от захвата:
заместители выбираются тем же ранжированием за всё время, в которое
нельзя войти быстро (RFC-0002, раздел 4.2), а добиться исключения
*живого* якоря атакующий не может — для этого сам якорь должен
промолчать, будучи выбранным, сквозь многодневную публичную остановку
(об оговорке с eclipse-атакой см. 5.1). Эффект продвижения строго
положителен для живучести: на текущей цепочке фильтрация поднимает в
якорный пул два ключа-ветерана и впускает в недавний пул два
по-настоящему новых доменных ключа (Приложение A).

### 4.6 Почему пол — и почему 10

Допущенная популяция равна 16 и монотонно сокращается (смерти плюс
растущий порог `height/100`; входа нет — раздел 7). Выбору нужны 7
различных ключей без автора, поэтому 8 — жёсткий минимум
завершаемости; `BAN_POOL_FLOOR = 10` держит запас в два ключа сверх
него, чтобы естественное выбывание после кристаллизации не заморило
выбор немедленно, и при этом позволяет удалить все семь ныне
подозрительных ключей по мере накопления свидетельств (16 − 6 = 10;
седьмой ждёт, пока рост живого ключа или выпадение мёртвого из-под
порога не освободит место). Пол ограничивает только *новые*
исключения; минимум из 3.4 — предохранитель, который не пробивает
никакая комбинация исключений и выбывания.

### 4.7 Форки, откаты и разделения сети

`B(H)` — чистая функция префикса цепочки, поэтому протокола не
требует: откат (`replace_block`) её перевычисляет (реализации должны
сбрасывать кэши исключений в точности как сегодня сбрасываются
`SignersCache` и `HealingCache`). При разделении, вылеченном с обеих
сторон (остаточный риск RFC-0002), две ветви могут кристаллизовать
разные множества исключённых; при воссоединении проигравшая ветвь
откатывается, и её исключения испаряются вместе с её блоками.
Исключения, таким образом, не добавляют нового механизма образования
форков — они лишь следуют за той ветвью, которая побеждает по
существующим правилам. Один эффект второго порядка отметим честно: на
каждой ветви ключи другой стороны молчат и кристаллизуются как
неявившиеся всюду, где были выбраны, так что долгое разделение
превращается во взаимные исключения на проигравших ветвях. Поскольку
выживают только исключения победившей ветви, итоговая цепочка
исключает максимум те ключи, что действительно отсутствовали *на
ней*, — это определение, работающее как задумано, но операторам
многодомных парков стоит знать: именно присутствие по обе стороны
разделения (сознательная подпись на обеих) не даёт их собственным
ключам кристаллизоваться ни на одной.

## 5. Соображения безопасности

### 5.1 Исключение как оружие

Чтобы добиться исключения ключа-жертвы, атакующий должен вызвать
вылеченное окно, в котором жертва была выбрана и не создала ничего.
Молчание нельзя подделать — подписи суть обычные распространяемые
блоки, — поэтому единственный вектор: лишить жертву связности на весь
таймаут: устойчивая eclipse- или DoS-атака на одного конкретного
оператора длительностью пять с лишним дней, во время которой
остальная сеть на виду останавливается и лечится, кристаллизация
громко объявляется (3.7), и затронуты могут быть максимум ключи,
выбранные для этого одного окна. Атака дорогая, медленная, видимая, а
её выигрыш (удаление одного ключа из выбора, который атакующий не
контролирует) мал. Она принимается как остаточный риск; смягчение то
же, что и для остаточных рисков RFC-0002, — операционная видимость и
реакция людей.

### 5.2 Самозащита бесплатна

Выбранный подписант избегает исключения, делая ровно то, для чего
подписант существует: держит узел, который подписывает. Льготный
срок — лечащий таймаут: узел ключа может лежать днями внутри
выбранного окна без последствий, лишь бы вернулся до закрытия окна.
Никакого нового поведения от операторов не требуется.

### 5.3 Взаимодействие с намайниванием

Множество исключённых нигде не облегчает и не вознаграждает вход:
допуск по-прежнему требует накопления `height/100`, якорное
ранжирование по-прежнему за всё время, а недавний пул не изменён,
кроме пропуска доказанных трупов. Пробел с кулдауном продлений
(RFC-0002, раздел 5.1) по-прежнему стоит закрыть отдельно; именно он —
причина отказа от воскрешения полным блоком (4.3), так что его
закрытие позволит будущему RFC пересмотреть необратимость.

### 5.4 Спам и DoS

По сети не передаётся ничего нового, и никакая новая валидация не
запускается пиром: работа кристаллизации ограничена одним вычислением
множества на полный блок, кэшируема и выполняется всеми узлами
одинаково.

## 6. Обратная совместимость

Это жёсткий форк, и — в отличие от RFC-0002, активировавшегося на
цепочке, которая не могла двигаться по старым правилам, — этот
активируется на **живой** цепочке. Узлы без него будут вычислять
выбранные наборы и резервные пулы с исключёнными ключами, принимать
подписывающие блоки, которые обновлённые узлы отвергают, и отвергать
блоки, которые обновлённые принимают, — и отколются на первом же
пост-активационном окне, где наборы разойдутся. Поэтому:

* `BAN_ACTIVATION_HEIGHT` установлена в **22 529** — вершину цепочки
  на момент релиза. Нулевой запас в норме был бы безрассудством, но
  на этой цепочке сегодня он безопасен по той же структурной причине,
  что и в RFC-0002, раздел 6.1: деградировавший режим продвигается
  примерно на один полный блок за лечащий таймаут, так что первый
  пост-активационный выбор наступит через дни после релиза, а поднятая
  минимальная версия пира (ниже) отключит устаревшие узлы прежде, чем
  они смогут пойти за расходящейся ветвью. Окно полного блока
  22 526 — ещё открытое на момент релиза — по построению сохраняет
  свой доактивационный выбор и пул.
* Релиз — **0.10.0**, минимальная версия пира поднимается до
  `>= 0.10.0` (`version_compatible` в `src/p2p/network.rs`), чтобы
  устаревшие узлы отключались, а не беззвучно форкались, — в точности
  как предписывает RFC-0002, раздел 6.

Вылеченные окна между `HEALING_ACTIVATION_HEIGHT` и
`BAN_ACTIVATION_HEIGHT` **считаются свидетельством**: их неявившиеся
кристаллизуются задним числом (в порядке цепочки, пол применяется на
момент каждого закрывающего блока), но *применяются* только в выборах
и пулах выше высоты активации. Свидетельство июльской остановки 2026
года, таким образом, не пропадает: первый же пост-активационный выбор
уже исключает четыре ключа, которые её вызвали. `CHAIN_VERSION` не
меняется; единственный шлюз — высота активации.

## 7. Ограничения: пул по-прежнему только сокращается

Этот RFC убирает мёртвый вес; живого веса он не добавляет. У
допущенной популяции нет входа: новому ключу нужно ~225 созданных
блоков, чтобы выбираться, но подписывать он не может, пока не
выбираем, а 225 полных блоков — нереалистичный путь. Исключения
ускоряют движение пула к полу в 10 ключей, из которых сегодня максимум
9 доказуемо или правдоподобно живы, и примерно половина из них
принадлежит одному оператору. Конечное состояние одного этого RFC —
меньший, чистый, более централизованный выбор, который в основном
фиксируется за минуты, а не за дни. Запас хода, а не вечность — ровно
в формулировке RFC-0002, раздел 7.

Последствия, принятые осознанно:

* Как только пол зажмёт популяцию на 10, оставшиеся мёртвые ключи
  исключить нельзя; окна, выбравшие ≥4 из них, по-прежнему лечатся
  путём RFC-0002. Пол обменивает часть остаточных остановок на
  разнообразие выбора.
* Ошибочно исключённый оператор не может вернуться к подписанию ни по
  одному действующему правилу, потому что и свежий ключ не может
  достичь допуска. Этот RFC этого не ухудшает — это существовавший до
  него пробел входа, — но необратимость делает его ощутимым. Будущий
  RFC-0004 (вход в допуск подписантов) — то дополнение, которое эта
  конструкция предполагает; закрытие пробела с кулдауном продлений
  также вновь откроет вариант воскрешения, отвергнутый в 4.3.
* Если выбывание однажды победит даже временные возвраты минимума на
  момент выбора (порог допуска пройдут менее 8 ключей вообще), выбор
  заморится независимо от исключений. Это финал храповика допуска, вне
  рамок этого документа, и точка, где оставшийся путь — RFC-0001 или
  форк-преемник.

## 8. Эталонная реализация

Ожидаемая форма (всё в существующих файлах, без новых модулей):

1. `src/commons/constants.rs`: три константы из 3.1.
2. `src/blockchain/chain.rs`:
   * `BansCache` по аналогии с `HealingCache`: накопительный список
     исключений с индексом кристаллизации у каждой записи,
     пополняемый инкрементально при добавлении полных блоков,
     сбрасываемый в `replace_block`.
   * Кристаллизация на каждом закрывающем полном блоке: тест
     вылеченного окна (уже реализован для адаптивного таймаута),
     `N(F)` обходом окна, `E(H)` одним SQL-подсчётом, проверка пола,
     упорядочение.
   * `get_block_signers`: пропуск ключей из `B(G)` внутри обхода (той
     же формы, что пропуск по `minimum_block_count`); предвычисление
     множества временного возврата при `|E \ B| < BAN_DRAW_MIN`.
     Заметьте: обход будет итерировать дольше, когда исключены тяжёлые
     ключи (их исторические блоки становятся мёртвыми выборками);
     существующее смягчение `tail / 13` применяется без изменений.
   * `get_healing_pool`: фильтрация исключённых (через `NOT IN` или
     пост-фильтр) в обоих ранжированиях.
   * Строка лога и событие UI кристаллизации (3.7).
3. Тесты: вывод неявившихся (поздний подписант засчитан, автор
   закрывающего блока засчитан, невылеченное окно не кристаллизует
   ничего); срез пола в порядке убывания; детерминизм выбора с
   исключениями; временный возврат на момент выбора (ровно до
   `BAN_DRAW_MIN`, обратный порядок); фильтрация и продвижение пула;
   ретроактивная кристаллизация ниже высоты активации с применением
   выше; сброс при откате.

## 9. Открытые вопросы

1. `BAN_POOL_FLOOR`: предложено 10 (жёсткий минимум 8 плюс запас в
   два). Более низкий пол раньше исключает больше трупов; более
   высокий сохраняет разнообразие, но дольше оставляет мёртвые ключи
   выбираемыми.
2. Порядок временного возврата на момент выбора: последние исключённые
   первыми (предложено: последними видены живыми, значит, вероятнее
   всего ещё существуют) или самые лёгкие первыми (наименьший вес в
   выборе, если действительно мертвы).
3. Следует ли кандидата, пропущенного полом, вместо этого откладывать
   и кристаллизовать автоматически при появлении места, экономя одно
   лишнее вылеченное окно. (Предложено: нет — отсутствие состояния
   стоит одного окна.)
4. Включать ли закрытие кулдауна продлений (RFC-0002, раздел 9,
   пункт 4) в один из ближайших релизов. (Уже не открыто:
   `BAN_ACTIVATION_HEIGHT = 22 529`, релиз — `0.10.0` с
   `version_compatible >= 0.10.0`, см. раздел 6; исправление кулдауна
   в него не входит.)
5. Вход в допуск подписантов (будущий RFC-0004): без него конечное
   состояние этого RFC — пул размером с пол без пути преемственности.

## Приложение A. Разобранный пример (реальные данные цепочки, высота 22 529)

Полный блок `F` = 22 521 (автор `BD3E…`), добыт 2026-07-05 03:58:50
UTC. Выбранные подписанты: `BE46…`, `BBA0…`, `6E24…`, `9A46…`,
`4116…`, `B7FB…`, `1502…`. Выбранные подписи 22 522–22 524 (`BE46…`,
`BBA0…`, `6E24…`) приходят за шесть минут; дальше — ничего. Лечащий
таймаут истекает 2026-07-10 03:58:50; лечащая подпись 22 525
(`9B89…`, якорь) приходит 2026-07-11 18:29:24 и фиксирует окно.
Закрывающий блок `F'` = 22 526 (полный, автор `6E24…`,
2026-07-11 20:21:44).

Окно вылечено (`9B89…` ∉ drawn(F)). Авторы блоков 22 522–22 526:
`{BE46…, BBA0…, 6E24…, 9B89…}`. Следовательно:

```
N(F) = { 4116…, 1502…, 9A46…, B7FB… }
```

Кристаллизация на 22 526, кандидаты в порядке убывания числа
созданных блоков: `4116…` (2 098), `1502…` (1 411), `9A46…` (1 326),
`B7FB…` (229). `|E(22 526)| = 16`; исключение всех четырёх оставляет
12 ≥ `BAN_POOL_FLOOR`, поэтому кристаллизуются все четыре. (Их
последние созданные блоки: 2026-06-26, 2026-06-26, 2026-06-26 и
2025-09-14 — вердикт механизма совпадает с реальностью, видимой
операторам.)

Влияние на следующий выбор: исключённые ключи несли ~27% веса
допущенных; оставшиеся подозрительные (`22DF…`, `CEE7…`, `B830…`)
несут ~5,6%. Ожидаемое число мёртвых ключей на выбор из семи падает с
~3 (остановка как подброс монеты) до заметно меньше 1 (редкие
остановки, каждая из которых кристаллизует новое свидетельство).

Влияние на резервный пул последующих полных блоков:

```
A = { BE46…, 9B89…, BBA0…, 6E24…, A980…, 2A02…, BBE5… }   (продвинуты 2A02…, BBE5…)
R = { 268E…, BC10…, 22DF…, 9B84…, 70AB… }                 (входят 9B84…, 70AB…)
```

Оба продвинутых якоря — ключи-ветераны (1 213 и 683 блока за всё
время); два новых участника недавнего пула — активные доменные
ключи: тот самый путь входа, который описывает RFC-0002, раздел 4.3,
теперь не вытесняемый трупами.
